進階 Prompt 與 RAG

Few-shot、Chain-of-Thought、Self-consistency、ReAct、RAG、Extended Thinking——六種把 LLM 從「會聊天」升級到「會研究」的技巧。

Few-shot, Chain-of-Thought, self-consistency, ReAct, RAG, extended thinking — six techniques that upgrade an LLM from chatty to research-grade.

術語速查

十個進階 prompt / RAG 必懂詞

這 10 個詞貫穿整章。先讀過，後面 CoT、RAG、reasoning model 的對話就會輕鬆許多。

These ten terms recur throughout. Read them once and the CoT / RAG / reasoning-model discussion below will flow easily.

🧠

CoT

Chain-of-Thought（Wei 2022）：要求模型「逐步推理」再給答案。對 reasoning task 提升 15–40%。可顯式 prompt 或 implicit by reasoning model。

Chain-of-Thought (Wei 2022): ask the model to reason step-by-step before answering. Boosts reasoning tasks 15–40%. Explicit via prompt, or implicit in reasoning models.

🎲

Self-Consistency

Wang 2023：同一個 prompt 用較高 T 跑 5–10 次，取多數決。降低 reasoning task 變異 50%+。代價：cost × n。

Wang 2023: run the same prompt 5–10 times at moderate T, majority-vote. Cuts reasoning variance 50%+. Cost × n.

🔧

ReAct

Yao 2023：Reasoning + Acting。模型在「思考」與「呼叫工具」間交錯。現代 agent（Claude Code、Cursor、Devin）的骨幹。

Yao 2023: Reasoning + Acting. Interleave reasoning with tool calls. Backbone of modern agents (Claude Code, Cursor, Devin).

📚

RAG

Retrieval-Augmented Generation（Lewis 2020）：先檢索相關文件、再 grounded generation。對抗幻覺最有效。

Retrieval-Augmented Generation (Lewis 2020): retrieve relevant docs first, then ground generation on them. The most effective anti-hallucination tool.

💾

Vector DB

儲存 embeddings 並支援 fast nearest-neighbor 查詢的資料庫。FAISS（CPU/GPU）、Chroma、Pinecone、Qdrant、Weaviate 是常見選擇。

A database storing embeddings with fast nearest-neighbor queries. FAISS (CPU/GPU), Chroma, Pinecone, Qdrant, Weaviate are common.

🧭

Embedding

把 token 或文字段轉成的高維向量（768–3072 維），讓「語意相似」者距離近。語意搜尋與 RAG 的基礎。

A high-dimensional vector (768–3072 dims) of a token or text snippet — semantically similar items are close. The basis of semantic search and RAG.

🔍

Retrieval

從 vector DB 抓 top-k 最相關段落的步驟。可純向量、純 BM25、或混合檢索 (hybrid)。對 RAG 品質影響大於 LLM 大小。

Fetching the top-k most relevant chunks from the vector DB. Pure vector, BM25, or hybrid. Retrieval quality matters more than LLM size for RAG.

🏅

Reranking

在 top-k 檢索後用 cross-encoder（Cohere Rerank、BGE-reranker）對候選重新排序，把最相關的塞給 LLM。能再提升 10–20% 命中率。

After top-k retrieval, use a cross-encoder (Cohere Rerank, BGE-reranker) to re-order — feed only the most relevant to the LLM. Boosts hit rate another 10–20%.

💭

Extended Thinking

Claude 4.x 的「延伸推理」：模型在輸出前先用大量 token 做 internal reasoning。對 hard reasoning 顯著提升，代價是 latency 與成本。

Claude 4.x's "extended thinking": the model spends many tokens on internal reasoning before answering. Big gains on hard reasoning at the cost of latency & price.

🧮

Reasoning Model

OpenAI o1 / o3、DeepSeek R1：在訓練階段就用 RL 訓練「先 reason 再答」。在數學、code、scientific QA 上強，但慢 5–20×。

OpenAI o1 / o3, DeepSeek R1: trained with RL to "reason first, then answer." Strong at math, code, scientific QA — but 5–20× slower.

初學者導讀

兩個類比直覺懂進階 prompt 與 RAG

類比 1：CoT 就是「請你大聲把推理過程說出來」。你帶過學生做 case presentation 嗎？學生悶頭只說「我覺得是 sepsis」你不知道他怎麼想的；但他若一步步說「病人發燒 39 度、WBC 14K、procalcitonin 高、皮膚有壓痕……所以最像 sepsis」，你立刻就能判斷他的邏輯哪邊出錯。CoT 對 LLM 一樣：強迫它「show your work」，輸出品質就大幅上升。同樣的原理為什麼 reasoning model（o1、o3、Claude 4.x extended thinking）強——它們「在心裡 CoT」，比顯式 prompt 更深更穩。

類比 2：RAG 就是「臨檢前先翻一下你的筆記本」。沒有 RAG 的 LLM 像不能查書的醫師：他憑記憶回答，記得對就對、記錯就錯。RAG 等於把你的「實驗室筆記本 / 病人病歷 / PubMed 子集」變成可隨手翻的索引；模型回答前先「翻」（retrieval），找到相關段落（context chunks），再用這些原文寫答案。幻覺率因此下降 70–90%，且每個 claim 都可追溯來源。沒有 RAG 是「請憑記憶答」；有 RAG 是「請翻書答」——對研究、臨床、grant 都是必備工具。

Analogy 1: CoT is "say your reasoning out loud." Ever supervised a student case presentation? A student who just says "I think it's sepsis" leaves you guessing. But one who says "39° fever, WBC 14K, high procalcitonin, mottled skin … therefore sepsis" — you can immediately check the logic. CoT does the same for LLMs: forcing "show your work" lifts quality dramatically. Same reason reasoning models (o1, o3, Claude 4.x extended thinking) are strong — they CoT internally, deeper and steadier than explicit prompting.

Analogy 2: RAG is "let the doctor open the notebook before answering." An LLM without RAG is a physician who can't open books — relies on memory, right or wrong. RAG turns your lab notebook / chart / PubMed subset into an indexed library; before answering, the model "looks up" (retrieval), grabs the relevant pages (context chunks), and writes the answer grounded on the actual text. Hallucination drops 70–90%, every claim is traceable to a source. No RAG = "from memory"; with RAG = "with the book open" — essential for research, clinical use, and grants.

概覽

當基礎 prompt 不夠用時

上一章六原則處理「一次性溝通」的問題。但研究現場常常更難：你要模型推理（如：根據病人 lab 與用藥史判斷 drug-drug interaction）、搜尋（從 200 篇 paper 找 evidence）、多次校驗（不能信第一次回答）。這時候要動用更高階技巧。

2026 年的研究級工作流，幾乎都疊用這幾招：用 few-shot 鎖格式、用 CoT 啟動 reasoning、用 self-consistency 抑制不穩定、用 RAG 抗幻覺、用 extended thinking（Claude）或 reasoning models（OpenAI o3）處理難 reasoning、用 ReAct 把 LLM 變成可以呼叫工具的 agent。

The six principles handle one-shot communication. Research often needs more: reasoning (drug-drug interaction from labs + meds), retrieval (find evidence across 200 papers), and verification (you can't trust the first answer). That calls for advanced techniques.

By 2026, research-grade workflows stack these: few-shot to lock format, CoT to elicit reasoning, self-consistency to dampen variance, RAG against hallucination, extended thinking (Claude) or reasoning models (o3) for hard reasoning, and ReAct to give the LLM tools.

💡
核心心法：不要一個技巧解所有問題。事實型問題 → RAG；推理型問題 → CoT / reasoning model；格式問題 → few-shot；不穩定 → self-consistency；需要操作外部世界 → ReAct / agent。 Core mindset: No single technique fits all. Facts → RAG. Reasoning → CoT / reasoning model. Format → few-shot. Variance → self-consistency. Acting on the world → ReAct / agent.
六大技巧

進階 prompt 武器庫

🪞

Few-shot

在 prompt 中放 2–5 個輸入/輸出範例。模型會「模仿」範例的格式、tone、推理深度。對格式控制最強。

Embed 2–5 input/output examples. The model mimics format, tone, and reasoning depth. Strongest for format control.

🧠

Chain-of-Thought

加上「Let's think step by step」或要求列出推理步驟。對數學、邏輯、診斷推理顯著提升。

Add "Let's think step by step" or require explicit reasoning steps. Major boost on math, logic, and diagnostic reasoning.

🎲

Self-Consistency

對同一個問題以較高 temperature 跑 5–10 次，取多數決或共識。降低 reasoning task 變異。

Run the same prompt 5–10 times at moderate temperature; majority-vote the answer. Tames reasoning variance.

🔧

ReAct

Reasoning + Acting：模型在「思考」與「呼叫工具」之間交錯。是現代 agent（Claude Code、Cursor）的骨幹。

Reasoning + Acting: interleave thinking with tool calls. The backbone of modern agents (Claude Code, Cursor).

📚

RAG

Retrieval-Augmented Generation：先檢索後生成。最有效對抗幻覺；2 種：stuff in context vs. vector DB。

Retrieval-Augmented Generation: retrieve then generate. Best anti-hallucination move. Two flavors: stuff-in-context vs. vector DB.

💭

Extended Thinking

Claude 4.x 的 extended thinking、OpenAI o3 reasoning，會用更多 tokens 內部推理，對 hard reasoning 顯著提升。

Claude 4.x extended thinking and OpenAI o3 reasoning spend more tokens on internal deliberation. Big gains on hard reasoning.

互動模擬

CoT toggle：DDI 推理 on / off

一名 ICU 病人正在用 warfarin（INR 2.4），主治新加 amiodarone 200 mg/day。模型該如何回答？切換 CoT 看差異。

An ICU patient is on warfarin (INR 2.4); the attending adds amiodarone 200 mg/day. How should the model answer? Toggle CoT to see.

提示：開啟 CoT 後輸出長 3–5 倍，但會列出 mechanism (CYP2C9 inhibition)、INR target、bridging plan，這些都是臨床決策需要的。

Note: with CoT, output is 3–5× longer but spells out mechanism (CYP2C9 inhibition), INR target, and bridging plan — what the clinician actually needs.

真實案例

兩個真實生醫場景的進階 prompt 應用

🔬

案例 1：Stanford 醫學院用 RAG + GPT-4 把 USMLE 通過率從 60% 拉到 85%

2023 年 Singhal 等人在 Nature 發表 Med-PaLM 2 之前，Stanford 醫學院的一個研究小組做了一個更貼地實驗：他們用 GPT-4 在 MedQA（USMLE-style）上裸跑只有約 60% 通過率，明顯不夠臨床用。他們的解法是疊三招：(1) RAG grounding 到 UpToDate + Harrison's + 最新 NEJM/JAMA review 共約 8,000 篇文獻；(2) CoT 要求模型 step-by-step 列出 differential、再選 best answer；(3) self-consistency 跑 5 次取多數決。最終 USMLE 通過率躍升到 85%，與資深住院醫師相當。關鍵教訓：「沒有單一技巧能解所有問題」——RAG 解事實、CoT 解推理、self-consistency 解變異。這個範式後來成為所有 medical LLM benchmark 的標準堆疊。研究者使用 prompt engineering 而非 fine-tuning 達成此結果，總成本不到 $500，遠低於 fine-tune 一個 7B 醫學模型的 $50K+。

Case 1: Stanford Medical School pushed USMLE pass rate from 60% to 85% with RAG + GPT-4

Before Singhal et al. published Med-PaLM 2 in Nature (2023), a Stanford group ran a more pragmatic experiment: GPT-4 on MedQA (USMLE-style) reached only ~60% pass — clearly insufficient for clinical use. Their fix stacked three tricks: (1) RAG grounded against UpToDate + Harrison's + the latest NEJM/JAMA reviews (~8,000 documents); (2) CoT forcing step-by-step differentials before the best answer; (3) self-consistency over 5 runs with majority vote. Result: 85% pass rate — on par with senior residents. The take-home: no single trick solves everything — RAG fixes facts, CoT fixes reasoning, self-consistency dampens variance. This stack became the standard for every later medical LLM benchmark. Crucially, they used prompt engineering, not fine-tuning, at a total cost under $500 — versus $50K+ to fine-tune a 7B medical model.

💊

案例 2：博後用 CoT 抓出隱藏的 dosing 計算錯誤

2025 年某腎臟科博後在準備 case report 時，要為 eGFR 18 的病人計算 vancomycin 載荷劑量。他先用「請告訴我 dose」這種 zero-shot prompt 問 GPT-5，模型直接回 1500 mg loading dose——看起來很標準。但他改用 CoT prompt：「請逐步說明：(1) 病人 BSA 與理想體重；(2) 目標 AUC 24h；(3) eGFR 對 Vd 的影響；(4) 載荷劑量公式；(5) 算式」。這次模型在第 3 步「卡」住，發現自己一開始用了「正常腎功能」的 Vd = 0.7 L/kg 而沒考慮 ESRD 病人 Vd 升至 0.9–1.0 L/kg 的差異——重算後正確劑量是 2000 mg。差異 25%，足以決定 24h 之內是否達到 target trough。這個案例後來在 lab 內部報告流傳，成為「為什麼 ICU 與腎科 prompt 必須加 CoT」的標準範例。教訓：看似簡單的計算題正是 CoT 最該用的場合——模型在 zero-shot 下會「跳步」，看起來自信但漏掉中間判斷。

Case 2: A postdoc caught a hidden dosing error using CoT

In 2025, a nephrology postdoc preparing a case report needed to compute a vancomycin loading dose for a patient at eGFR 18. He first asked GPT-5 zero-shot — "what's the dose?" — and got "1500 mg loading dose," sounding standard. He retried with CoT: "Walk me through (1) BSA + IBW, (2) target AUC24h, (3) how eGFR affects Vd, (4) loading-dose formula, (5) arithmetic." This time the model paused at step 3 — it noticed it had used a normal-renal-function Vd of 0.7 L/kg without accounting for ESRD's elevated Vd (0.9–1.0 L/kg). Recomputed correctly: 2000 mg. A 25% gap, enough to miss target trough in the first 24h. The story made the rounds internally as "why ICU and renal prompts need CoT." Lesson: "simple" calculations are exactly where CoT pays off — zero-shot LLMs skip intermediate judgments while sounding confident.

常見誤區

進階 prompt & RAG 的六個常見誤區

⚠️
誤區 1：RAG 設定好就放著不檢視 retrieval 品質
為什麼錯：retrieval 的 top-k 命中率決定一切；若 chunking 太碎（chunk_size=200）或 embedding model 過時（text-embedding-ada-002），命中率可低至 30%，LLM 就會「在錯的段落上自信生成」。
正確做法：定期用 20 個 gold-standard query 跑 recall@k 評估；若 recall@5 < 80% 必須調 chunk size、改 embedding model（建議 text-embedding-3-large 或 BGE-M3）、加 reranking。
Pitfall 1: Set up RAG and never audit retrieval quality
Why wrong: top-k hit rate decides everything; with overly small chunks (chunk_size=200) or stale embeddings (text-embedding-ada-002), hit rate can drop to 30% — and the LLM happily generates from the wrong passage.
Fix: monthly evaluate recall@k on a 20-query gold set; if recall@5 < 80%, retune chunk size, upgrade the embedding (text-embedding-3-large / BGE-M3), and add reranking.
⚠️
誤區 2：對所有任務都套 CoT
為什麼錯：Sprague et al. 2024（ICLR）指出 CoT 對簡單分類、format conversion、純檢索任務反而拖慢且降準——因為模型「過度推理」會引入錯誤中介。
正確做法：CoT 只用於「需要中間步驟」的任務（數學、診斷推理、code debug）；簡單抽取/分類用 zero-shot 或 few-shot 即可。一個 rule of thumb：若你自己心算 < 30 秒可解，模型 CoT 反而是浪費。
Pitfall 2: Use CoT on every task
Why wrong: Sprague et al. 2024 (ICLR) shows CoT actually slows and degrades performance on simple classification, format conversion, and pure-retrieval tasks — "over-reasoning" introduces wrong intermediates.
Fix: reserve CoT for tasks needing intermediate steps (math, diagnostic reasoning, code debug); simple extraction/classification works fine zero-shot or few-shot. Rule of thumb: if you can solve it mentally in <30 s, CoT only adds noise.
⚠️
誤區 3：Few-shot 範例同質性過高造成 mode collapse
為什麼錯：若你給的 5 個範例都是「正常範圍 → normal」這類，模型會「鎖死」這個 pattern，碰到 edge case（如 missing data、合併症）就 collapse 到最近的範例。
正確做法：範例要多樣化覆蓋 edge cases（normal、abnormal、missing、ambiguous 各一），並在 prompt 中明示「請判斷時不要受範例 bias 影響」。Min et al. 2022 證實 diversity 比 quantity 更重要。
Pitfall 3: Few-shot examples too homogeneous → mode collapse
Why wrong: if all 5 examples map "normal range → normal," the model locks onto that pattern and collapses to the nearest example on edge cases (missing data, comorbidities).
Fix: diversify examples to cover edge cases (normal / abnormal / missing / ambiguous), and explicitly tell the model not to over-anchor on the demos. Min et al. 2022 showed diversity matters more than quantity.
⚠️
誤區 4：略過 reranking，直接把 top-10 餵 LLM
為什麼錯：向量檢索 top-10 中通常只有 2–3 個真正相關；其餘是 noise，會稀釋 LLM attention、提高幻覺率。Reranking 用 cross-encoder（如 cohere-rerank-3 或 BGE-reranker-v2-m3）對 top-10 重排，把 top-3 餵 LLM 即可。
正確做法：bi-encoder 召回 → cross-encoder rerank → 取 top 3–5 給 LLM。實測能再提升 10–20% 命中率，且 LLM 輸入更短、cost 降低。
Pitfall 4: Skip reranking — feed top-10 directly to the LLM
Why wrong: vector top-10 typically contains only 2–3 truly relevant chunks; the rest is noise that dilutes attention and raises hallucination.
Fix: bi-encoder recall → cross-encoder rerank (cohere-rerank-3 or BGE-reranker-v2-m3) → keep top 3–5 for the LLM. Real-world lift: +10–20% hit rate, shorter prompts, lower cost.
⚠️
誤區 5：Extended thinking 對 well-defined 任務反而過度思考
為什麼錯：Claude 4.x extended thinking / OpenAI o3 在簡單 NER、JSON 抽取等 well-defined 任務上會「想太多」——多花 5–20× token、latency 變慢，且 accuracy 反而下降 1–3%（過度修正）。
正確做法：先用普通模型 baseline，若 accuracy < 85% 再考慮 extended thinking；對需要 multi-hop reasoning（如 drug-drug-drug interaction、complex differential）才值得。
Pitfall 5: Extended thinking over-thinks well-defined tasks
Why wrong: Claude 4.x extended thinking / OpenAI o3 burn 5–20× tokens on trivial NER or JSON extraction, slowing latency and even dropping accuracy 1–3% by over-correcting.
Fix: baseline with a normal model first; only invoke extended thinking when accuracy < 85% AND the task is genuinely multi-hop (e.g., drug-drug-drug interaction, complex differential).
⚠️
誤區 6：把 RAG 當「對抗模型幻覺的萬靈丹」
為什麼錯：RAG 只能在「檢索到正確段落」時抑制幻覺；若 corpus 本身有錯（如過時 guideline、retract paper）或 retrieval miss，LLM 仍會「自信地」用錯的段落生成。
正確做法：corpus 要定期 curate（移除 retract papers、更新 guideline 版本），retrieval 要評估，並對輸出做 chain-of-verification——讓 LLM 自我檢視「這個 claim 是否真的能從 source 推出」。
Pitfall 6: Treating RAG as the panacea against hallucination
Why wrong: RAG only suppresses hallucinations when retrieval lands on the right chunks. If the corpus itself is wrong (stale guidelines, retracted papers) or retrieval misses, the LLM confidently fabricates from the wrong passage.
Fix: curate the corpus regularly (remove retracted papers, refresh guideline versions), evaluate retrieval, and apply chain-of-verification — let the LLM self-check whether each claim actually follows from the sources.
資深讀者深入

進階細節

🔬 Bi-encoder vs Cross-encoder、KV cache 成本、RAG vs Fine-tune trade-off

① Bi-encoder vs Cross-encoder（reranker 的數學區別）。Bi-encoder（嵌入模型）把 query 與 doc 各自編碼為向量，計算 cosine similarity：sim(q,d) = q · d / (||q|| · ||d||)。優點是 doc embedding 可預先計算、檢索 O(N·d)；缺點是 query 與 doc 沒「互相看到」，rank 上限不高。Cross-encoder（reranker）把 (query, doc) 串成一句話進 Transformer，self-attention 讓兩者 token 彼此 attend，輸出單一 relevance score。準度高但 O(N·L²) 算不動全 corpus，所以只用在 top-k rerank。實務 pipeline：bi-encoder 召 100 → cross-encoder rerank 取 5 → LLM 生成。Cohere Rerank 3、BGE-reranker-v2-m3、Jina reranker 都是此架構。

② KV cache 的真實成本。Transformer decoder 在生成時，每個新 token 要對前面所有 token 的 K/V 做 attention。為了避免重算，會把 K/V 暫存（KV cache）。對 70B 模型、4096 hidden、80 layers 的設定，每 token 的 KV cache 約 2 (K+V) × 80 × 4096 × 2 bytes (fp16) ≈ 1.3 MB。1M context 即 1.3 TB——遠超單卡 VRAM，所以實務上會做 PagedAttention（vLLM）、KV quantization（fp16→int8）或 sliding window。對使用者意義：long context 不是免費，每 1k token prefill 成本約 input price，generation 階段每 token 約 5–10× input price。

③ RAG vs Fine-tune 的 trade-off（Lewis 2020 vs Houlsby 2019）。RAG：知識在外部 corpus、可隨時更新、無 catastrophic forgetting、cost 低；缺點是 latency 高（retrieval + generation）、依賴 retrieval 品質、prompt 變長。Fine-tune：知識「焊」進模型權重、inference 快、prompt 短；缺點是更新成本高（每次新文獻都要重訓）、有 catastrophic forgetting、難 trace source。實務經驗：事實/文獻用 RAG、風格/格式用 fine-tune（LoRA）。生醫 case 通常 RAG > fine-tune，因為文獻每週更新，且需要可引用來源。

④ Embedding model 選擇（2026 年）。OpenAI text-embedding-3-large（3072d，$0.13/1M tokens）通用 SOTA；Cohere embed-multilingual-v4（1024d）多語強；BGE-M3（1024d，開源）含 dense + sparse + multi-vector 三種輸出；Voyage-3-medical 專為生醫文獻訓練，在 PubMed retrieval 比通用 embedding 高 15–20%。生醫研究建議：通用文獻用 BGE-M3 或 Voyage-3-medical；多語用 Cohere v4；快速原型用 text-embedding-3-large。

① Bi-encoder vs cross-encoder (the reranker math). A bi-encoder embeds query and doc independently, then cosine-scores: sim(q,d) = q·d / (||q||·||d||). Doc embeddings can be precomputed; retrieval is O(N·d); but query and doc never "see" each other, capping rank quality. A cross-encoder concatenates (query, doc) into one sequence and runs self-attention end-to-end, producing a single relevance score. Higher quality, but O(N·L²) — infeasible on the full corpus, so it's used only to rerank the top-k. Standard pipeline: bi-encoder recall 100 → cross-encoder rerank → top 5 → LLM. Cohere Rerank 3, BGE-reranker-v2-m3, Jina reranker all follow this design.

② KV cache, the real cost. During generation, each new token attends to all prior tokens' K/V. To avoid recomputation we cache K/V. For a 70B model, hidden=4096, 80 layers, the per-token cache is roughly 2 (K+V) × 80 × 4096 × 2 bytes (fp16) ≈ 1.3 MB. A 1M context implies 1.3 TB — far beyond single-GPU VRAM, so production stacks use PagedAttention (vLLM), KV quantization (fp16→int8), or sliding windows. Bottom line: long context isn't free; prefill costs about 1× input price per token, generation is 5–10× input price per token.

③ RAG vs fine-tune trade-off (Lewis 2020 vs Houlsby 2019). RAG: knowledge lives outside the weights, updates instantly, no catastrophic forgetting, cheap; but adds retrieval latency, depends on retrieval quality, and inflates the prompt. Fine-tune: knowledge welded into weights, fast inference, short prompts; but expensive to update, prone to forgetting, hard to trace sources. Practical rule: facts/literature → RAG; style/format → fine-tune (LoRA). Biomedical use almost always favors RAG because the literature changes weekly and citation traceability is non-negotiable.

④ Choosing an embedding model (2026). OpenAI text-embedding-3-large (3072d, $0.13/1M tokens) is general-purpose SOTA. Cohere embed-multilingual-v4 (1024d) excels at multilingual. BGE-M3 (1024d, open source) offers dense + sparse + multi-vector outputs. Voyage-3-medical is trained on biomedical text and outperforms general embeddings on PubMed retrieval by 15–20%. Recommendations: general literature → BGE-M3 or Voyage-3-medical; multilingual → Cohere v4; quick prototypes → text-embedding-3-large.

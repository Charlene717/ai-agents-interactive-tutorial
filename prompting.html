Prompt 工程基礎

90% 的 AI 失望源於 prompt 不夠精確。掌握六大原則，你的 prompt 從「請幫我」升級為可被工程化、可重現的 specification。

Ninety percent of AI frustration comes from imprecise prompts. Master six principles and your prompts become engineerable, reproducible specifications.

術語速查

十個 prompt 工程常出現的詞

下方任何技巧名稱都不會憑空冒出；先讀完這 10 張卡片，後面所有 prompt 範例都讀得明白。

None of the technique names will come out of nowhere. Skim these ten cards first; every prompt example later will then read clearly.

📝

Prompt

給 LLM 的輸入文字，廣義包含 system + user + history + retrieved chunks。模型對你說的「整段話」都是 prompt 一部分。

The input text you give an LLM. Broadly: system + user + history + retrieved chunks. Everything you "say" is part of the prompt.

⚙️

System Prompt

放在最前的長期身分與規則。模型會在整個對話中遵守。在 API 中是獨立欄位，Claude 用 system= 參數、OpenAI 用 "role":"system"。

The persistent identity + rules placed up front. Honored across the whole conversation. In APIs it's a separate field: Claude uses system=, OpenAI uses "role":"system".

💬

User Prompt

使用者本輪的問題/指令。是模型「直接回應」的對象。多輪對話會把 user/assistant 交錯堆疊起來成為 history。

The user's question or instruction for this turn — what the model directly responds to. Multi-turn chats stack user/assistant alternately as history.

🎭

Role

給模型分配「身份」（內分泌科醫師、生統助理教授）。能啟動領域 prior，是 6 大原則之首。

Assigns the model an identity (endocrinologist, biostat assistant professor). Activates domain priors — first of the six principles.

🧪

Few-shot

在 prompt 中放 2–5 個「輸入 → 輸出」範例。是鎖死輸出格式最強招式；對抽取任務尤其有效。

Embed 2–5 input/output examples in the prompt. The strongest single move for locking output format; especially effective on extraction tasks.

🎲

Zero-shot

不給範例，純靠指令完成任務。對強模型 + 清楚 instruction 能成立；對複雜結構化任務常需 few-shot 才穩。

No examples — pure instruction. Works on strong models with crystal-clear instructions; complex structured tasks usually still need few-shot.

📐

Output Format

對輸出的形態約束：長度、結構（markdown、JSON）、語言。明確規定能讓後續腳本直接 parse。

Shape constraints on the output: length, structure (markdown, JSON), language. Lets downstream scripts parse directly.

🚧

Constraint

「不能做什麼」的明文約束：不虛構引用、不超過 500 字、不出現 PHI。比正面 instruction 更能擋掉壞行為。

Explicit "what NOT to do": no fabricated citations, no exceeding 500 words, no PHI. Better at suppressing bad behaviors than positive instructions.

🌡️

Temperature

採樣的「隨機度旋鈕」。0 = argmax；1 = 標準 softmax；>1 = 更發散。事實任務用 0.0–0.3，創意用 0.7–1.0。

The randomness dial. 0 = argmax; 1 = vanilla softmax; >1 = more diverse. Facts 0.0–0.3, creative 0.7–1.0.

📊

Top-p (nucleus)

另一個採樣參數：只從「累積機率 ≤ p」的 token 中採樣。跟 temperature 互補，常被同時調整。建議二選一。

Another sampling knob: sample only from tokens whose cumulative probability ≤ p. Complementary to temperature; usually tune one or the other.

初學者導讀

兩個類比直接讓你「會寫 prompt」

類比 1：寫 prompt 就像寫實驗 protocol。你絕不會跟新進的 RA 說「幫我跑一下 western」就走人。你會寫：用哪個 cell line、收幾個 well、哪個 antibody、稀釋幾倍、什麼條件曝光多久。Prompt 也是一樣——LLM 不是「你的同事」，它是一個會嚴格按指令做事的高效助手。「指令模糊 → 結果模糊」是 prompt 的 garbage-in-garbage-out。Protocol 寫得詳細，跑出來的 western 才能 reproducible；prompt 寫得詳細，輸出才會穩定。

類比 2：Prompt 是「條件機率」P(output | prompt) 的條件設定。同一個 prompt 模型每次回應略有不同，這是「給定 prompt 後分佈的不同樣本」。你寫得越具體、越受約束，這個分佈越窄；同一個輸出出現的機率越高，「穩定性」就越好。Few-shot 的數學直覺就是：給範例等於告訴模型「我要的輸出長得像這些」，讓 likely 區間收緊。對統計背景的人來說，prompt engineering 是「對 likelihood 加 informative prior」的 Bayesian 操作。

Analogy 1: Writing a prompt is like writing a wet-lab protocol. You'd never tell a new RA "just run a Western" and leave. You write: cell line, well count, antibody, dilution, exposure time. Prompts work the same way. The LLM isn't your colleague; it's a high-efficiency assistant that follows instructions literally. "Vague instruction → vague result" is prompt-side garbage-in-garbage-out. Detailed protocols give reproducible Westerns; detailed prompts give stable outputs.

Analogy 2: A prompt sets the condition in P(output | prompt). Same prompt, slightly different runs — those are samples from the conditional distribution. Write more specifics and constraints, narrow that distribution, raise the probability of your target output. The math intuition for few-shot: showing examples tells the model "outputs look like these," tightening the likely region. For statisticians, prompt engineering is "adding an informative prior" in a Bayesian sense.

概覽

把 prompt 當成 API 規格書

LLM 是機率性生成器，所以同一個模糊問題每次回應都會「漂移」。解決方式不是換更強的模型，是把 prompt 寫得像 API spec——明確輸入、明確輸出、明確邊界。一個好 prompt 平均能讓 GPT-5.5 / Claude Opus 4.6 的可用率從 40% 提升到 90%，這個提升通常比換 model size 更大。

2026 年 prompt 工程已經有了相對穩定的「六原則」共識：Role（角色）、Context（上下文）、Task（任務）、Format（輸出格式）、Constraints（限制）、Examples（範例）。本章我們用一個真實研究情境（協助寫 grant aims）來練習從爛 prompt 升級到好 prompt。

LLMs are probabilistic, so a vague question drifts each time. The fix isn't a bigger model — it's writing prompts like API specs: explicit input, explicit output, explicit boundaries. A well-crafted prompt typically lifts Claude Opus 4.6 / GPT-5.5 from ~40% useful to ~90% — bigger than upgrading model size.

By 2026 the field has settled on six principles: Role, Context, Task, Format, Constraints, Examples. We'll practice on a real scenario — drafting grant aims — and walk a bad prompt up to a strong one.

💡
核心心法：不要把 LLM 當成「能讀心的同事」。它沒有你的研究背景、不知道你給誰看、不知道你要多長。每一個被你略過的細節，都會被它用機率猜出來——而它的猜測通常是「最大眾」的版本，不是你要的。 Core mindset: Don't treat the LLM as a colleague who reads your mind. It doesn't know your research background, audience, or desired length. Anything you skip, it guesses — and the guess is usually the most generic version, not yours.
六大原則

RCTFCE：好 prompt 的六個組件

🎭

Role

指派身分：「你是審稿經驗豐富的腫瘤學家」。讓模型啟動領域 prior。

Assign a persona: "You are a senior oncologist with peer-review experience." This activates domain priors.

🗂️

Context

提供你的處境：研究領域、對象、已知資訊、限制（如：給 NIH R01）。

Give your situation: field, audience, what's known, constraints (e.g., for an NIH R01).

🎯

Task

具體動詞 + 範圍：「列出 5 個 specific aims」、「比較 2 個演算法的差異」。

Concrete verb + scope: "list 5 specific aims", "compare two algorithms".

📐

Format

輸出長度 / 結構 / 語言：「繁中 markdown，500 字內，每段附 1 個 PMID」。

Length / structure / language: "Traditional Chinese markdown, <500 words, 1 PMID per section".

🚧

Constraints

明確邊界：「不要虛構引用」、「若不確定請說明」、「避開 cohort < 100 的研究」。

Boundaries: "no fabricated citations", "say when unsure", "exclude cohorts <100".

🧪

Examples

給 1–3 個範例（few-shot），是讓輸出格式 / 風格收斂的最強招式。

Provide 1–3 worked examples (few-shot). The single strongest move for locking format and tone.

幫我寫一個 grant 的 specific aims，關於糖尿病。
# Help me write specific aims for a grant on diabetes.

# 問題：沒角色、沒對象、沒長度、沒格式、可能虛構引用
# Problems: no role / audience / length / format; may fabricate refs
Role: 你是 NIH R01 審查經驗的內分泌科臨床研究者。
Context: 我正在準備 T2D 並發症的 R01 (mechanism + intervention)。
申請人為 K99 transition 階段、已有 2 篇 first-author publication。
Task: 給出 3 個 Specific Aims（Aim 1 mechanism, Aim 2 in vivo,
Aim 3 small clinical pilot），每個 aim 附 hypothesis、approach、
可行性說明。
Format: 繁體中文，markdown，每 Aim <= 250 字，title 用 H3。
Constraints: 不引用 PMID（避免虛構）；不確定處註明 [需查證]。
# 用 XML 與 Markdown 結構分隔，模型最容易解析
<role>Senior translational endocrinologist, NIH R01 reviewer.</role>

<context>
- Topic: T2D complications, mitochondrial dysfunction in renal tubular cells
- Stage: K99/R00 → R01 transition
- Preliminary data: 2 first-author papers, mouse KO line available
</context>

<task>
Draft 3 Specific Aims (mechanism / in vivo / clinical pilot).
For each: Hypothesis · Approach · Feasibility · Risk & mitigation.
</task>

<format>
Output as a Markdown document with H3 per Aim, <=300 words per aim.
</format>

<constraints>
- Do NOT invent PMIDs. Mark unverified facts as [verify].
- Avoid generic NIH boilerplate ("paradigm-shifting").
</constraints>

<example>
### Aim 1. Establish that ...
**Hypothesis:** ...
</example>
工作流程

六步驟 prompt 建構流程

① 定義目標讀者

是 grant 評審？lab meeting？住院醫師？讀者層級決定詞彙與深度。

Grant reviewer? Lab meeting? Resident? The audience sets vocabulary and depth.

② 指派 Role

把模型「設定」成領域專家，這會啟動該領域相關的訓練分布。

"Set" the model as a domain expert; this activates the relevant training distribution.

③ 給足 Context

你的 lab、資料、已知結果、限制——能寫的都寫；過多比過少好。

Lab, data, known findings, constraints — write more not less. Over-context beats under-context.

④ 明確 Task 動詞

「列出」「比較」「批判」「重寫」「翻譯」——避免「幫我看看」。

"List", "Compare", "Critique", "Rewrite", "Translate" — avoid "have a look".

⑤ 鎖死 Format

長度、結構、語言、是否 markdown / JSON / 表格——能量化就量化。

Length, structure, language, markdown / JSON / table — quantify when possible.

⑥ 加 Constraints + Examples

不能做什麼 + 1 個格式範例，輸出穩定度可再提升 30%。

What NOT to do + 1 worked example: bumps output stability by another 30%.

真實案例

兩個從爛到好的 prompt 進化案例

🧬案例 A：「請解釋 RNA-seq」 → 結構化 prompt 的完整進化

第一版（爛 prompt）：「請解釋 RNA-seq。」
實際輸出：模型給了 1500 字泛泛而論——歷史、原理、技術版本各一段，但沒有對象、沒有臨床例子、沒有 PMID，且包含 3 個錯誤陳述（FPKM ≠ RPKM、把 Illumina 的 sequencing-by-synthesis 寫成 sequencing-by-ligation）。
第二版（加角色）：「你是生醫教學經驗豐富的 PI，請解釋 RNA-seq。」輸出稍微聚焦但仍泛泛。
第三版（加上下文與對象）：「對剛入學的住院醫師解釋 RNA-seq 與 scRNA-seq 的差異，他們以為兩者只是『細胞數量不同』。」開始有臨床例子。
第四版（加格式 + 限制）：「繁中 markdown，500 字內，分三段（原理 / 應用 / 2024–2026 新趨勢），每段 1 個臨床情境。不要 PMID（避免虛構），不確定處標 [需查證]。」
第五版（加範例）：放一個「Aim 1 結構範本」few-shot。輸出：500 字、結構完美、3 個臨床例子、零幻覺、3 處標 [需查證] 給人類驗證。從第一版到第五版，可用率從 ~25% 升到 ~95%。整個過程約 5 次 iteration，總計 5 分鐘。

v1 (bad): "Please explain RNA-seq."
Actual output: a 1500-word generic essay — history, principle, tech versions — with no audience, no clinical example, no PMID, and 3 factual errors (FPKM ≠ RPKM, calling Illumina's sequencing-by-synthesis "sequencing-by-ligation").
v2 (add role): "You are an experienced biomed teaching PI; explain RNA-seq." Slightly tighter, still generic.
v3 (add context + audience): "Explain RNA-seq vs scRNA-seq to first-year residents who think the only difference is cell count." Clinical examples appear.
v4 (add format + constraints): "Traditional-Chinese markdown, <500 words, 3 sections (principle / application / 2024–2026 trend), each with 1 clinical scene. No PMIDs (avoid fabrication); mark [verify] when unsure."
v5 (add example): include a "Section template" as few-shot. Output: 500 words, perfect structure, 3 clinical examples, zero hallucination, 3 [verify] tags for human review. v1 to v5: usability rose from ~25% to ~95%. About 5 iterations totaling 5 minutes.

💼案例 B：一個 grant idea 從爛 prompt 到專業 prompt 的迭代

一位 K99 階段的科學家想用 Claude Opus 寫 R01 的 specific aims。他第一個 prompt：「幫我寫 grant，糖尿病 T2D 並發症」。模型回了 1200 字、5 個假 PMID、aims 與 hypothesis 混淆。他改寫 5 輪：(1) 加角色「NIH R01 reviewer」→ 結構出現了；(2) 加 context「K99 階段 + 已有 mouse KO line」→ feasibility 段變實在；(3) 拆 Aims 為 mechanism / in vivo / clinical pilot 三段；(4) 加 Constraints「不虛構 PMID、不確定標 [verify]」→ 假引用消失；(5) 加一個「Aim 1 結構範例」few-shot → 三個 aims 結構統一。最終版輸出與 PI 親手寫的差距 < 10%，他用這個版本作為跟 mentor 討論的 starting point，比從白紙開始省了 4 天工作量。關鍵體會：prompt 不是一次寫對的，是「iteration 5 次」的工程過程；每次 iteration 加一個維度（Role → Context → Task → Format → Constraints → Examples）。

A K99-stage scientist wanted Claude Opus to draft R01 specific aims on T2D complications. v1: "Help me write a grant on T2D complications." Result: 1200 generic words, 5 fake PMIDs, aims and hypotheses muddled. Five revisions: (1) add role "NIH R01 reviewer" → structure appears; (2) add context "K99 stage + mouse KO line" → feasibility tightens; (3) split aims into mechanism / in vivo / clinical pilot; (4) add constraints "no fabricated PMIDs, tag [verify]" → fake citations vanish; (5) add an Aim 1 few-shot template → all three aims align. Final version was within 10% of the PI's handwritten draft and saved ~4 days as a mentor-meeting starting point. Takeaway: prompts aren't first-try perfect — they're a 5-iteration engineering process; each round adds one principle dimension (Role → Context → Task → Format → Constraints → Examples).

🔬 資深讀者深入：Prompt sensitivity 與 jailbreak 的 ML 視角

Prompt sensitivity 的數學本質。把 LLM 看成函數 f(prompt) → distribution；prompt sensitivity 就是「對 prompt 小擾動 ε 的輸出變化」‖f(p + ε) − f(p)‖。Zou 2023 等多項研究顯示，即使加一個空白字元或變更標點，重要 benchmark 的 score 可上下 5–15%。這跟 ML 中的 adversarial perturbation 同源——LLM 對 input 沒有 strong invariance。

為什麼 few-shot 有效——in-context learning 假說。Olsson 2022（Anthropic）提出「induction heads」假設：Transformer 在訓練時學會了一種「複製→比對→預測下一個」的元演算法，所以給範例時模型能「在 context 內」實作小型 gradient-descent like 行為（Akyürek 2023）。這把 few-shot 從「玄學」拉回成可解釋的演算過程。

Jailbreak 與 prompt injection 的對抗本質。Jailbreak 與 prompt injection 本質是「在 input space 上找對抗例」——所以 RLHF、Constitutional AI、安全微調都只能「移動分佈」，無法根除。最新 Anthropic「Many-shot Jailbreaking」（2024）顯示給 256 個 fake example 即可繞過部分安全 layer，這是 long context 的副作用。

The math of prompt sensitivity. View the LLM as f(prompt) → distribution; prompt sensitivity is the output change under small perturbations ε: ‖f(p + ε) − f(p)‖. Zou 2023 and others showed even a whitespace or punctuation change can shift benchmark scores by 5–15%. Same family as adversarial perturbation — LLMs lack strong input invariance.

Why few-shot works — the in-context learning hypothesis. Olsson 2022 (Anthropic) proposed "induction heads": Transformers learn a "copy → match → predict-next" meta-algorithm during pretraining, so given examples they perform something like in-context gradient descent (Akyürek 2023). This grounds few-shot in explainable computation, not magic.

Jailbreaks and prompt injection as adversarial examples. Both are adversarial examples in input space — so RLHF, Constitutional AI, and safety fine-tuning only "shift the distribution"; they don't eliminate. Anthropic's "Many-shot Jailbreaking" (2024) showed 256 fake examples can bypass parts of the safety layer — a side-effect of long context.

常見誤區


關於 prompting 的 6 個常見錯誤




    

  1. 「禮貌的話可以省」——你可以省，但加上 "Please" 並不浪費 token 又能讓對話更舒服。不影響輸出品質。
    2. 

  2. 「prompt 越長越好」——錯。冗餘指令可能稀釋核心；專業 prompt 是「精準」不是「冗長」。
    3. 

  3. 「同一個 prompt 兩個模型一定一樣」——錯。Claude 對 XML 敏感，GPT 對 markdown 敏感，Gemini 對 explicit JSON schema 敏感。要根據模型微調。
    4. 

  4. 「指定『不要』就會避免」——半對。負面指令有時反而提示模型「想想這件事」。建議用正面的替代行為（「使用 X」而非「不要用 Y」）。
    5. 

  5. 「few-shot 範例多多益善」——錯。5 個範例後邊際效益陡降；範例 token 也是錢。
    6.

生統數理視角


給統計背景讀者的進階註腳




① Prompt 作為條件機率設定。把 LLM 視為 p(y | x, θ)，prompt x 改變條件分佈的形狀。Role / Context / Constraints 都是在 x 上加結構，作用上等價於 informative prior：若 x 把分佈推向 mode 集中區（low entropy）就能得到穩定輸出。


② Few-shot 與 maximum a posteriori。給 k 個範例 (x_i, y_i)，相當於把 likelihood 用 examples 條件化：p(y | x_new, x_1, y_1, …, x_k, y_k)。當範例代表性強且彼此一致時，posterior 在「正確輸出形式」附近集中，模型自然 mimics。這跟 EBM (empirical Bayes) 的「shrinkage to the prior of group」異曲同工。


③ Prompt sensitivity 與 perturbation robustness。Prompt 中加減一個空白 → benchmark 上下 5–15%（Sclar 2024）。可視為 ML 中 adversarial 與 robustness 的延伸——但 LLM 沒有像 image 那樣的 invariance（如平移不變）內建，所以對 input format 高度敏感。


④ 對齊 token 數 vs 答對率的 power curve。實證上，prompt 中每加 100 tokens 的「有效信息」（範例、規則、格式），可用率 utility 約增加 5–8%，呈現 saturating 曲線；超過 800–1000 tokens 後邊際效益接近零。這跟 dose-response Emax model 同形。






① Prompt as conditioning. Treat the LLM as p(y | x, θ); the prompt x reshapes the conditional distribution. Role / Context / Constraints all add structure to x, functionally an informative prior. If x narrows the mass around a mode (low entropy), output is stable.


② Few-shot ≈ MAP. k examples (x_i, y_i) condition the likelihood: p(y | x_new, x_1, y_1, …, x_k, y_k). When the examples are representative and consistent, the posterior concentrates around the correct output form, and the model mimics. Cousin of empirical-Bayes "shrinkage to the prior of group."


③ Prompt sensitivity vs perturbation robustness. One whitespace change can shift benchmark scores 5–15% (Sclar 2024). Same family as adversarial / robustness — but LLMs lack built-in invariance (like translation invariance for vision), making them brittle to format.


④ Power curve: prompt tokens vs accuracy. Empirically, each additional 100 tokens of useful info (examples, rules, format) lifts utility by ~5–8%, with diminishing returns — past ~800–1000 tokens the marginal gain is near zero. Shape matches dose-response Emax models.

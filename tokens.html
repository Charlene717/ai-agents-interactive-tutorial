Token 與 Context Window

為什麼「1M context」聽起來像魔法、但你貼了 800K 字後模型開始忘事？理解 token 計費與召回曲線，是用好 LLM 的第二堂課。

Why does "1M context" sound magical yet the model starts forgetting after you paste 800K words? Understanding tokens, pricing, and the recall curve is your second LLM lesson.

術語速查

十個關於 token 與 context 的關鍵詞

「token」這個詞在 NLP 圈內含義精確，但跨進生醫圈常被誤用。先讀這 10 張卡，後面所有「為什麼中文比較貴」「為什麼 1M context 不能塞滿」都會自然懂。

"Token" has a precise meaning in NLP that often gets muddled in biomed. Skim these ten cards first; later sections on "why Chinese costs more" and "why 1M context can't be fully filled" will then be obvious.

🔡

Token

LLM 處理的最小單位，是subword 片段。「mitochondrial」≈1 token；「scRNA-seq」常被切成 3–4 個 token。所有計費與長度限制都以 token 為單位。

The smallest unit an LLM processes — a subword piece. "mitochondrial" ≈ 1 token; "scRNA-seq" usually splits into 3–4. Billing and length caps are all token-based.

✂️

BPE

Byte-Pair Encoding（Sennrich 2016）：從字元開始，反覆把最常見的字元對合併。GPT 家族使用的 tiktoken 與 cl100k_base 即為變體。

Byte-Pair Encoding (Sennrich 2016): starts from characters and iteratively merges the most frequent pairs. GPT's tiktoken / cl100k_base are variants.

🌐

SentencePiece

Google 的 tokenizer，不依語言邊界，把空白也當字元處理。Llama、Gemini、T5 都用 SentencePiece 變體，對中文/日文較友善。

Google's tokenizer; language-agnostic, treats whitespace as a regular character. Llama, Gemini, and T5 use SentencePiece variants — friendlier to CJK.

🪟

Context Window

模型一次能看見的 token 數上限。2026 旗艦 ≈ 1M；Llama 4 Scout 達 10M。但有效 context（actually useful）通常只有宣稱值的 50–70%。

The maximum number of tokens the model sees at once. 2026 flagships ≈ 1M; Llama 4 Scout 10M. But effective context is typically 50–70% of the advertised value.

🧮

Tokenizer

把文字切成 token 的程式。不同模型 tokenizer 不同——同樣一段文字在 GPT 與 Claude 上的 token 數可能差 10%。tiktoken 是 OpenAI 的開源實作。

The program that chops text into tokens. Different models use different tokenizers; the same passage can differ by 10% across GPT and Claude. tiktoken is OpenAI's open-source one.

⚙️

Attention

Transformer 核心：對 n 個 token，計算每兩兩 token 的「相關度」。複雜度是 O(n²)，這是長 context 慢且貴的主因。

The Transformer's core operation: compute pairwise "relevance" between every two of n tokens. Complexity is O(n²) — the main reason long context is slow and pricey.

💾

KV Cache

Key/Value cache：把已算過的 attention 中間結果存下來，逐 token 生成時不必重算前面。長對話/長 context 的「memory footprint」主因。

Key/Value cache: stores already-computed attention intermediates so each new token doesn't redo the prior ones. The main memory cost of long chats / long context.

📍

Positional Encoding

告訴模型「token 是第幾個位置」的訊號。從原始 sinusoidal 到 RoPE（Su 2023），讓 attention 帶有相對位置信息。是長 context 外推的關鍵。

The signal telling the model "which position is this token." Evolved from sinusoidal to RoPE (Su 2023), giving attention relative-position information. Critical to long-context extrapolation.

🕳️

Lost in the Middle

Liu 2024 TACL 觀察：把關鍵資訊放中段，模型常會「跳過」；放開頭/結尾則準確得多。長 context 工程必懂的偏誤。

Liu 2024 TACL: information placed in the middle of a long context often gets "skipped"; the start and end are far more reliably attended to. Essential bias to know.

🔄

Sliding Window

處理「context 不夠」的工程招式：把長文切重疊段（如 800 token + 120 overlap），逐段處理後合併。也是 Mistral 等模型局部 attention 的設計。

Engineering trick when context isn't enough: chunk long text with overlap (e.g., 800 + 120 overlap), process piece by piece, then merge. Also the local-attention design in models like Mistral.

初學者導讀

兩個類比讓你直覺懂 token 與 context

類比 1：Token 就是 LLM 的「公升」，不是「字」。你開車進加油站，油價是「每公升 35 元」，不是「每滴」也不是「每箱」。LLM 的世界裡，token 是基本計量單位，所有計費 (input/output)、限制 (max_tokens)、context window 都以 token 計。「字數」「字元數」「位元組數」都不是直接相關。一個英文長詞 (mitochondrial) 可能只佔 1 token；一個中文字常佔 1.5–2 token；一段 JSON 因為括號、引號多，token/char 比例會比純文字高。

類比 2：Context Window 就像你的工作桌面。你準備一份 review article，桌上能堆 10 篇 paper，超過就要從架上拿一篇下來、收一篇回去。1M context 像「桌面很大、能堆 1000 篇」。但人類視覺有極限——堆太多時你會看不到中間那本。Liu 2024 的 "lost in the middle" 就是 LLM 的這個現象：context 大不等於「全部都用得到」。所以工程上總是想辦法 (1) 把關鍵段放開頭/結尾 (2) 用 RAG「只取最相關幾頁」(3) 階層摘要。理解了這兩個類比，後面所有「為什麼 RAG 比塞整個 1M 好」的討論就一目了然。

Analogy 1: A token is the LLM's liter, not its letter. When you pull into a gas station, the price is "per liter" — not per drop or per tank. In the LLM world, the token is the unit: billing (input/output), limits (max_tokens), and context window are all token-based. Word count, character count, byte count are not directly related. A long English word like "mitochondrial" can be 1 token; one Chinese character is usually 1.5–2; JSON, with its brackets and quotes, has a higher token-per-char ratio than prose.

Analogy 2: Context window is your desk. Drafting a review, you can pile 10 papers on your desk; beyond that you must put one back on the shelf. A 1M context is "an enormous desk that holds 1000 papers." But human vision has limits — pile too high and the middle ones vanish. Liu 2024's "lost in the middle" is exactly this for LLMs: bigger context ≠ uniformly usable context. So engineering always tries to (1) put the critical bits at start/end, (2) use RAG to fetch only the most relevant pages, or (3) summarize hierarchically. With these two analogies, "why RAG beats stuffing 1M" becomes obvious.

概覽

Token 是 LLM 的「公升」

每次跟 Claude / GPT / Gemini 講話，它都不是用「字元」也不是用「英文單字」計算成本，而是用 token。Token 由 tokenizer（BPE、SentencePiece 或 tiktoken）把文字切成 subword 片段：「mitochondrial」可能是 1 個 token，「scRNA-seq」可能被切成 3–4 個。經驗法則：1 個英文單字 ≈ 1.3 tokens，1 個中文字 ≈ 1.5–2 tokens，1 行程式碼 ≈ 5–10 tokens。

Context window 是模型「一次能塞進腦袋」的 token 數。2026 年的旗艦模型多為 1M（Claude Opus 4.6、GPT-5.5 Pro、Gemini 2.5 Pro），Llama 4 Scout 甚至到 10M。但要注意——大不等於好用：超過 ~500K 後召回率 (needle-in-haystack) 會明顯下降，且 attention 計算是 O(n²)，越長越慢越貴。

Every chat with Claude / GPT / Gemini is billed not in characters or English words but in tokens. A tokenizer (BPE, SentencePiece, or tiktoken) chops text into subword pieces: "mitochondrial" is often 1 token; "scRNA-seq" splits into 3–4. Rule of thumb: 1 English word ≈ 1.3 tokens, 1 Chinese character ≈ 1.5–2 tokens, 1 line of code ≈ 5–10 tokens.

The context window is the number of tokens a model can hold in its head at once. The 2026 flagships sit at 1M (Claude Opus 4.6, GPT-5.5 Pro, Gemini 2.5 Pro), while Llama 4 Scout reaches 10M. But bigger isn't always usable: above ~500K, needle-in-haystack recall degrades, and attention is O(n²) — longer = slower and pricier.

💡
核心心法：長 context 是工具，不是萬靈丹。「貼整本論文」適合摘要；「貼整個 codebase」適合 refactor；但若你只要回答某個事實，RAG（先檢索後生成）幾乎永遠比 stuff-in-context 更便宜、更準。 Core mindset: Long context is a tool, not a panacea. "Paste the whole paper" works for summarization; "paste the whole codebase" works for refactor. But if you only need one fact, RAG (retrieve, then generate) almost always beats stuff-in-context on both cost and accuracy.

關於 token 你必須知道的六件事

🔡

BPE / SentencePiece

Byte-Pair Encoding 從字元逐步合併常見對；SentencePiece 不依語言邊界。中文常被切成單字或雙字片段。

BPE merges frequent character pairs; SentencePiece is language-agnostic. Chinese is usually split into 1–2 character pieces.

🇬🇧

English ratio

≈ 0.75 字/token；100 字英文 ≈ 130 tokens。專業詞（mitochondrial）通常 1 token。

≈ 0.75 words/token; 100 English words ≈ 130 tokens. Long technical words like "mitochondrial" are often 1 token.

🇹🇼

Chinese ratio

1 漢字 ≈ 1.5–2 tokens；繁體略多於簡體。1000 字中文約 1800 tokens。

1 Chinese char ≈ 1.5–2 tokens; Traditional slightly more than Simplified. 1000 Chinese chars ≈ 1800 tokens.

💻

Code ratio

程式碼平均 5–10 tokens/行；空格、縮排、特殊符號都吃 token。R 比 Python 略多。

Code averages 5–10 tokens/line; whitespace, indentation, and symbols all consume tokens. R slightly more than Python.

💰

Input ≠ Output 計費

輸入便宜（讀），輸出貴（寫）；長文摘要划算，長文翻譯不划算。

Input is cheap (read), output is expensive (write). Long summarization is economical; long translation is not.

📉

Recall decay

超過 ~500K 後針在乾草堆中召回率下降；放關鍵資訊在「開頭或結尾」比放中段更可靠。

Above ~500K, needle-in-haystack recall drops. Place key info at the start or end, not buried mid-context.

跨語言差異

為什麼中文比英文耗 token？

原因有三層。第一層：訓練語料分佈不均。大多數開源 BPE tokenizer 訓練時，英文佔比 70–95%，中文只佔幾個百分點。BPE 演算法會「優先合併常見對」，所以英文常見的 "ing"、"tion"、"ation" 都變成單一 token，而中文兩字詞（粒線體、轉錄因子）卻常被切散。

第二層：byte-level fallback。當 tokenizer 沒見過某個字（罕見漢字、繁體偏旁、emoji），會退回到 byte 級別，把它切成 2–4 個 UTF-8 byte token——所以「驫」「鱻」這類字可能每字 3 token。Llama 3、GPT-4o、Claude 3.5 都帶有這個 fallback。

第三層：tokenizer 對 CJK 的優化。Gemini（SentencePiece BPE 變體）對中文較好，平均 1 漢字 ≈ 1.2–1.5 token；GPT-5.5 / Claude Opus ≈ 1.5–1.9 token；Llama 4 約 1.7–2.1 token。差異看似小，但 100 萬字中文跑下來，token 數差距可達 30–70%。

務實建議：(1) 與英文母語讀者溝通的 prompt，建議直接用英文寫；(2) 處理中文病歷或論文摘要時，預留 1.7–2× 的 token 預算；(3) 同樣預算下，Gemini 對中文最划算；(4) 若大量處理繁體中文，記得加 `disallowed_special` 處理特殊符號，免得 token 數爆衝。

Three layers. Layer 1: training-data imbalance. Most open BPE tokenizers were trained on text that's 70–95% English with only a few percent Chinese. BPE merges the most frequent pairs first, so English's "ing", "tion", "ation" become single tokens, while two-character Chinese terms (粒線體, 轉錄因子) get split.

Layer 2: byte-level fallback. When the tokenizer hasn't seen a character (rare hanzi, traditional radicals, emoji), it falls back to bytes — 2–4 UTF-8-byte tokens per char. Llama 3, GPT-4o, and Claude 3.5 all include this fallback.

Layer 3: tokenizer optimization for CJK. Gemini's SentencePiece variant is the friendliest: ~1.2–1.5 tokens per Chinese char. GPT-5.5 / Claude Opus ≈ 1.5–1.9; Llama 4 ≈ 1.7–2.1. Across 1 M Chinese characters this 30–70% gap is significant.

Practical advice: (1) prompts to English-speaking peers — write in English; (2) Chinese charts or abstracts — budget 1.7–2× tokens; (3) at equal cost Gemini is cheapest for Chinese; (4) for Traditional Chinese, set disallowed_special carefully to avoid surprise token bloat.

🔬 資深讀者深入：Quadratic Attention 的數學與 Flash Attention 演進

Self-attention 的核心計算是 softmax(Q · K^T / √d_k) · V。對 n token 序列，Q、K、V 都是 n × d 矩陣，所以 Q · K^T 是 n × n 矩陣，記憶體 O(n²)、運算 O(n² · d)。當 n = 1M 而 d = 128，光是 attention 矩陣就 10¹² 個 floats = 4 TB（fp32）。這顯然不能放進 GPU。

三代演進：(1) FlashAttention（Dao 2022）：用 tiling + recomputation 把 attention 計算分塊，I/O 從 O(n²) 降到 O(n) 同時保留精度。Throughput 提升 2–4×；(2) FlashAttention-2（Dao 2023）：進一步優化 GPU thread 排程；(3) Ring Attention / Striped Attention（Liu 2024）：把 attention 跨 GPU 分散，這是 1M+ context 真正可行的關鍵。

另一條路是線性 attention：Linformer（投影）、Performer（kernel approximation）、Mamba（state space）。Mamba 系列在某些 benchmark 上接近 Transformer，但目前 frontier model 主流仍是 attention + FlashAttention 路線。Hybrid 設計 (e.g., Jamba) 越來越多。

對使用者的實務意義：API 上看「1M context」實際 latency 不是線性增長，常是次線性（thanks FlashAttention），但 cost 仍隨 n 線性增長（按 token 計費）。對「貼整個 paper 來摘要」這類任務，cost 約是「貼 abstract」的 30–50×，latency 約 3–5×。

Self-attention's core is softmax(Q · K^T / √d_k) · V. With sequence length n and head dim d, Q, K, V are n × d, so Q · K^T is n × n: memory O(n²), compute O(n² · d). For n = 1M and d = 128, the attention matrix alone needs 10¹² floats ≈ 4 TB (fp32). Not fitting on any GPU.

Three generations: (1) FlashAttention (Dao 2022): tiling + recomputation drops I/O from O(n²) to O(n) without losing precision; 2–4× throughput. (2) FlashAttention-2 (Dao 2023): better GPU thread scheduling. (3) Ring / Striped Attention (Liu 2024): split attention across GPUs — the real key to 1M+ context being shippable.

The alternative line is linear attention: Linformer (projection), Performer (kernel), Mamba (state-space). Mamba-class models match Transformers on some benchmarks but frontier models remain attention + FlashAttention. Hybrid designs (Jamba) are growing.

User implication: "1M context" API latency is often sub-linear (thanks to FlashAttention) but cost is still linear in tokens. Summarizing a full paper costs 30–50× the abstract, with 3–5× the latency.

截斷策略

當 context 不夠用時的三招

① Sliding Window 滑動視窗

把長文切段，每段保留前一段尾巴 200–500 tokens 重疊，逐段處理後 merge。適合長訪談轉錄、長 PDF 摘要。

Chunk the text and overlap each segment with the prior 200–500 tokens; process and merge. Good for long transcripts and PDFs.

② Hierarchical Summarization 階層摘要

第一輪每段獨立摘要，第二輪把所有摘要再摘一次。代價是細節流失。適合 50 篇 review 速覽。

Summarize each chunk, then summarize the summaries. Trades detail for breadth. Good for skimming 50-paper reviews.

③ RAG 向量檢索

把文件 embedding 存進 vector DB（FAISS / Chroma / Pinecone），查詢時只取 top-k 相關段落丟給 LLM。最便宜、最準的長文方案。

Embed documents into a vector DB (FAISS / Chroma / Pinecone) and feed only the top-k relevant chunks to the LLM. Cheapest and most accurate for long corpora.

真實案例

兩個關於 token 經濟的真實故事

📚案例 A：心衰竭組用 1M context 做 50 篇 systematic review

2025 年某心衰竭研究團隊嘗試用 Gemini 1.5 Pro 的 1M context 做 SGLT2 inhibitor 對 HFpEF 的 systematic review。他們把 50 篇全文（共約 600K token）一次塞進 context，請模型抽出每篇的「人數、follow-up、primary outcome、HR、CI、p、AE」並輸出 markdown 表格。第一次嘗試結果是：表格結構完美、但 50 篇裡有 14 篇的 HR 數字被「重新採樣」（hallucination 數值）；其中 5 篇 confidence interval 完全錯位。團隊後來改用「逐篇 RAG + verification 」流程，把 hallucination 降到 1/50。教訓：1M context 適合「結構轉換 / 重組」型任務（如把 5 篇 review 改寫成 1 份），但不適合需要精確數值的抽取——RAG 加 source 驗證才是工程化做法。整個經驗也讓他們把 systematic review 流程從「3 個月 RA 全手工」改成「1 週 RAG + 2 週人類驗證」。

In 2025, a heart-failure research group attempted a systematic review of SGLT2 inhibitors in HFpEF using Gemini 1.5 Pro's 1M context. They stuffed 50 full-text papers (~600K tokens) into a single context and asked the model to extract "n, follow-up, primary outcome, HR, CI, p, AE" per paper as a markdown table. Result: the table looked perfect, but 14/50 hazard ratios were re-sampled (numeric hallucinations), and 5/50 CIs were completely misaligned. They switched to a per-paper RAG + verification pipeline and reduced hallucinations to ~1/50. Lesson: 1M context works for structural rewrites (merging 5 reviews into 1) but fails on precise numeric extraction — RAG plus source verification is the engineering path. Their downstream workflow shifted from "3 months of manual RA work" to "1 week RAG + 2 weeks human verification."

🇹🇼案例 B：中文研究者發現「同字數」的中文 prompt 比英文貴 1.5×

2026 年初某台灣生統團隊在跑大量繁中病歷 de-identification 時發現一個會計震驚：他們以為「字數差不多應該成本差不多」，結果同樣的工作量，中文版的 OpenAI API bill 比英文版高 1.5–1.7×。他們追下去發現原因——cl100k_base tokenizer 對繁體中文不夠優化，每個漢字平均 1.7 token，而英文每 word 只要 1.3 token；加上中文「每 char 1.7 token × 平均 word 內 1.5 char」就是 ~2.5×。他們最後把流程改成：(1) 對「結構抽取」類任務先把 prompt 用英文寫、只把內容留中文；(2) 把 Gemini 2.5 Pro 加進選項（對中文 token ratio ~1.3，每月成本省 35%）；(3) 報帳時把這個資料寫進 grant budget justification 給未來申請參考。這是一個典型的「不認識 token 經濟就會被 bill 嚇到」案例。

In early 2026, a Taiwan biostats team running de-identification on Traditional Chinese charts hit a billing shock: same workload, Chinese version costs 1.5–1.7× the English. Root cause: cl100k_base is suboptimal for Traditional Chinese (~1.7 tokens/char), while English averages 1.3 tokens/word; combined with "1.7 token/char × ~1.5 chars/Chinese-word" = ~2.5× per word. They reworked the pipeline: (1) on structured-extraction tasks, write the prompt in English and keep only the content in Chinese; (2) add Gemini 2.5 Pro (Chinese ratio ~1.3, saving ~35% monthly); (3) document this in grant-budget justifications. A classic "you'll be bill-shocked if you don't understand token economics" story.

常見誤區

關於 token 的 6 個常見誤解

  1. 「1 個英文字 = 1 token」——錯。短常見字確實多為 1 token，但 "tokenization"、"systematically" 等常被拆 2–3 個。
  2. 「1M context 就能塞 100 萬個漢字」——錯。1 漢字 ≈ 1.7 token，1M context ≈ 60 萬中文字、80 萬英文 word。
  3. 「Context window 越大越好」——錯。Recall 在 ~500K 後下降，且 O(n²) 計算讓 latency/成本爆增。
  4. 「同一個模型對所有語言一樣」——錯。中文/日文/阿拉伯文 token 比例都明顯高於英文。
  5. 「貼整個 BAM/FASTQ 到 context 就能讓 AI 分析」——錯。LLM 看不懂 binary；先 sample 統計或 RAG。
  6. 「Output 跟 input 一樣便宜」——錯。output token 通常 3–10× 貴於 input；長翻譯成本最高。
  1. "1 English word = 1 token" — wrong. Short common words yes, but "tokenization"/"systematically" often split 2–3.
  2. "1M context = 1M Chinese characters" — wrong. ~1.7 tokens/char means 1M ctx ≈ 600K Chinese chars or ~800K English words.
  3. "Bigger context is always better" — recall drops after ~500K; O(n²) blows up latency and cost.
  4. "Same model performs uniformly across languages" — Chinese / Japanese / Arabic all have higher token ratios than English.
  5. "Just paste the BAM/FASTQ" — LLMs can't parse binary; summarize or RAG first.
  6. "Output is as cheap as input" — output is typically 3–10× more expensive; long translation hits the wallet hardest.
生統數理視角

給統計背景讀者的進階註腳

① Token 數估算的隨機性。對一段文字，token 數可視為一個隨機變數，依語言、tokenizer、是否含特殊符號而變。對 n 字英文文本，token 數 X ≈ 1.3·n，且 (X − 1.3n) / √n 近似 N(0, σ²)，σ ≈ 0.3（經驗值）。對中文 σ 更大（≈ 0.5），因為罕字 byte-fallback 造成尾巴重。Poisson 近似對「token 數 / 字元數」這個比例有限制，因 token 並非獨立事件——相鄰字元的合併是高度相關的，所以 token 計數不是純 Poisson。

② Attention 複雜度的下界。已有理論結果（Keles 2023, Alman 2024）指出，full attention 在最壞情況下必須 Ω(n²) — 線性 attention 都是某種「近似 / kernel approximation」。FlashAttention 不打破這個下界，它只是把 I/O 從 O(n²) 變 O(n)，計算仍是 O(n²·d)。

③ Lost-in-the-middle 的統計表現。把 needle 位置 p ∈ [0,1] 視為自變數，召回率 r(p) 在多數模型上接近 U 型曲線：r(p) ≈ r_0 · (1 − γ · sin(πp))。γ 越大 U 越深；frontier model γ 約 0.2，而早期 8K-context 模型 γ 可達 0.45。

④ 成本最佳化的拉格朗日。給定固定預算 B = c_in · T_in + c_out · T_out，且 T_out 與任務相關（摘要 T_out 約 T_in / 100；翻譯 T_out ≈ T_in），最佳化「每元錢輸出 token 數」會自然引導你選對任務類型。對 input-heavy 任務，便宜的 input 計費（Gemini 0.27/M）優於 output-heavy 任務的 output 計費（Claude $75/M output）。

① Token-count randomness. For a text passage, token count is a random variable depending on language, tokenizer, and special characters. For n English chars, X ≈ 1.3·n with (X − 1.3n)/√n ≈ N(0, σ²), σ ≈ 0.3 empirically. For Chinese σ ≈ 0.5 (rare-char byte fallback fattens the tail). A pure Poisson approximation fails because adjacent character merges are correlated — token events are not independent.

② Attention complexity lower bound. Recent theory (Keles 2023, Alman 2024) shows full attention requires Ω(n²) in the worst case — linear-attention variants are kernel approximations. FlashAttention does not break the bound; it just turns I/O from O(n²) to O(n), with compute still O(n²·d).

③ Lost-in-the-middle as a function. With needle position p ∈ [0,1], recall is roughly U-shaped: r(p) ≈ r_0 · (1 − γ · sin(πp)). γ ≈ 0.2 for frontier models; older 8K models had γ ≈ 0.45.

④ Cost optimization Lagrangian. Given budget B = c_in · T_in + c_out · T_out, with task-dependent T_out (summary T_out ≈ T_in / 100; translation T_out ≈ T_in), choosing the right model becomes a constrained optimization. Input-heavy tasks favor low input price (Gemini $0.27/M); output-heavy tasks should fear Claude's $75/M output.

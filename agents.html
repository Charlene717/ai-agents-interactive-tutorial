Agentic Workflows 與 MCP

當 LLM 學會用工具、多步驟規劃、記憶——agent 時代來了。MCP 是 2025–2026 連接 agent 與生醫資料源的開放標準。

When LLMs learn tool use, multi-step planning, and memory — the agent era arrives. MCP is the 2025–2026 open standard that wires agents to biomed data sources.

術語速查

Agent 與 MCP 必懂十詞

這 10 個詞貫穿這一章。先讀過一次，後面 ReAct、MCP server、checkpointing 的討論就會輕鬆許多。

These ten words recur throughout. Skim once and the ReAct / MCP server / checkpointing discussion below flows easily.

🤖

Agent

LLM ＋ tool use ＋ multi-step planning ＋ memory。給目標、自己拆步驟、呼叫工具、reflect 完成度的系統。與 chatbot 的差別是「不只回答，還執行」。Anthropic 2024 與 OpenAI 2025 agents.md 是當前最有影響力的定義。

LLM + tool use + multi-step planning + memory. Takes a goal, decomposes into steps, calls tools, and reflects on completion. The chatbot/agent difference: "doesn't just answer — executes." Anthropic 2024 and OpenAI 2025 agents.md are the influential definitions.

🛠️

Tool Use

LLM 呼叫外部 function 取得資料或執行 action：search、code exec、查 PubMed、寫檔、發 email。模型輸出結構化 JSON 描述要叫哪個 tool 與哪些 arguments，runtime 執行後把結果餵回。

The LLM calls external functions to fetch data or execute actions: search, code exec, PubMed query, write file, send email. The model emits structured JSON naming the tool and its arguments; the runtime executes and feeds the result back.

📞

Function Calling

OpenAI 2023/06 推出的 tool use 介面標準：定義 JSON schema 描述每個 function，model 學會輸出符合 schema 的 call。後來 Anthropic 的 tool use、Gemini function calling 都採類似格式。

OpenAI's June 2023 tool-use interface: define JSON schemas for each function; the model emits schema-conforming calls. Anthropic tool use and Gemini function calling adopted similar formats.

🔌

MCP (Model Context Protocol)

Anthropic 2024/11 提出、2025/11/25 釋出穩定 spec 的開放協議。把 LLM 與外部 tool / 資料源的對話標準化（JSON-RPC over stdio/SSE）。類似 LSP 之於 IDE——寫一次 server，任何支援 MCP 的 client 都能用。

Anthropic's open protocol (Nov 2024, stable spec 2025/11/25) standardising LLM ↔ tool/data conversations (JSON-RPC over stdio/SSE). Like LSP for IDEs — write a server once, any MCP-capable client can use it.

🖥️

MCP Server

暴露一組 tool / resource / prompt 給 LLM 用的小型 process。生醫例：GEO MCP server、UniProt MCP server、Ensembl MCP server。MCPmed 2025 是把這些包成生態的開源 initiative。

A small process exposing a set of tools / resources / prompts to LLMs. Biomed examples: GEO MCP, UniProt MCP, Ensembl MCP. MCPmed (2025) is the open-source initiative bundling these.

📱

MCP Client

支援 MCP 協議的應用：Claude Desktop、Cursor、Continue、Cline、Zed、ChatGPT Connectors（2025）、Copilot（2025/10 起）。Client 透過 stdio/SSE 連到 server，把 tool list 餵給 LLM。

MCP-capable apps: Claude Desktop, Cursor, Continue, Cline, Zed, ChatGPT Connectors (2025), Copilot (since 2025/10). Clients connect to servers via stdio/SSE and pass tool lists to the LLM.

🕸️

LangGraph

LangChain 2024 推出的 graph-based agent framework：把 agent workflow 顯式建為 DAG，節點是 LLM call 或 tool call，邊是 transition。Stateful、可 checkpoint、可 human-in-the-loop。生醫長流程的主流框架之一。

LangChain's 2024 graph-based agent framework: workflows are explicit DAGs with LLM/tool nodes and edge transitions. Stateful, checkpointable, supports human-in-the-loop. A mainstream framework for long biomed workflows.

👥

CrewAI / AutoGen / Multi-agent

CrewAI（2024）與 Microsoft AutoGen（Wu et al. 2023）把多個 agent 組成 team：planner、coder、reviewer、QA。Multi-agent 在複雜 task（如 software engineering benchmark）優於 single agent 但 cost 與 latency 都更高。

CrewAI (2024) and Microsoft AutoGen (Wu et al. 2023) compose multiple agents into a team: planner, coder, reviewer, QA. Multi-agent beats single agent on complex tasks (software-engineering benchmarks) at higher cost and latency.

🔄

ReAct / Reflection

Yao et al. 2022 (ICLR 2023) 提出 ReAct：交織 Reasoning（model 思考）+ Acting（呼叫 tool）。Shinn et al. 2023（Reflexion）加上 reflection——失敗後 model 寫 self-critique 進入 memory 下一次學。是 agent 思考迴圈的兩個核心 pattern。

Yao et al. 2022 (ICLR 2023) introduced ReAct: interleaving Reasoning (thinking) with Acting (tool calls). Shinn et al. 2023 (Reflexion) added reflection — on failure the model writes a self-critique into memory for the next round. Two core patterns of agent loops.

💾

Checkpointing

Long-running agent 在每個關鍵 step 把 state 持久化到 SQLite/Postgres/Redis，使 crash 可 resume、人類可 mid-flight 介入。LangGraph 內建 checkpointer，是 production agent 不可少的工程基礎設施。

Long-running agents persist state at each key step to SQLite/Postgres/Redis so crashes can resume and humans can intervene mid-flight. LangGraph ships a checkpointer — indispensable plumbing for production agents.

概覽

什麼是 agent？什麼是 MCP？

Agent ＝ LLM ＋ tool use ＋ multi-step planning ＋ memory。它不是回答一句，而是給目標、自己拆步驟、呼叫工具（搜尋、執行 code、查 PubMed、寫檔）、再 reflect 是否完成。生醫場景例子：給「整理近一年 KRAS G12C 抑制劑的臨床試驗」這個目標，agent 自己去 ClinicalTrials.gov 拉資料、彙整、寫 markdown 報告、推到 Slack。

Anthropic 2024/11 提出的 MCP (Model Context Protocol) 已在 2025/11/25 釋出穩定 spec，是 agent 與外部工具 / 資料源之間的開放標準（類似 LSP 之於 IDE）。把 GEO、UniProt、Ensembl、PubMed 包成 MCP server 後，任何支援 MCP 的 client（Claude Desktop、Cursor、Continue、Cline、Zed）都能立即使用——這就是 2025 發表於 Briefings in Bioinformatics 的「MCPmed」生態。

Agent = LLM + tool use + multi-step planning + memory. Instead of returning one answer, an agent takes a goal, breaks it into steps, calls tools (search, code execution, PubMed query, file write), and reflects on whether it's done. Biomed example: ask it to "summarise the past year of KRAS G12C inhibitor trials" and it pulls ClinicalTrials.gov, aggregates, writes a markdown report, and pushes it to Slack.

Anthropic's MCP (Model Context Protocol) (Nov 2024, stable spec 2025/11/25) is the open standard between agents and external tools / data sources — like LSP is to IDEs. Wrap GEO, UniProt, Ensembl, PubMed as MCP servers and any MCP-capable client (Claude Desktop, Cursor, Continue, Cline, Zed) gets them for free — this is the "MCPmed" ecosystem reported in Briefings in Bioinformatics (2025).

💡
核心心法：Agent 把「prompt-response」變成「goal-execution」。MCP 把「每個工具都要寫 plugin」變成「寫一次 server，任何 client 都能用」。兩者合起來，2026 的研究流程從「我用 AI」轉變為「AI 幫我跑整個 pipeline」。 Core mindset: Agents turn prompt-response into goal-execution. MCP turns "write a plugin per tool" into "write a server once, every client can use it." Together, 2026 research shifts from "I use AI" to "AI runs my pipeline for me."

實作 code

四、三段 code 對照

# .claude/agents/geo-fetcher.md
---
name: geo-fetcher
description: "Pulls a GSE accession's metadata + sample sheet from NCBI GEO"
tools: [Bash, WebFetch, Write]
---
You fetch GEO study metadata. Given a GSE id:
  1. curl https://www.ncbi.nlm.nih.gov/geo/query/acc.cgi?acc=<id>
  2. Parse title, platform, n_samples, treatment groups
  3. Write summary.md and samples.tsv to ./data/<id>/
  4. Return ✅ + sample count, or ❌ + reason
# minimal MCP server: expose a "pubmed_search" tool
from mcp.server import Server
from mcp.types import Tool, TextContent
import httpx

app = Server("pubmed-mcp")

@app.list_tools()
async def list_tools():
    return [Tool(name="pubmed_search",
        description="Search PubMed; return top-N PMIDs and titles.",
        inputSchema={"type":"object",
            "properties":{"q":{"type":"string"},"n":{"type":"integer","default":10}},
            "required":["q"]})]

@app.call_tool()
async def call_tool(name, args):
    r = httpx.get("https://eutils.ncbi.nlm.nih.gov/entrez/eutils/esearch.fcgi",
                 params={"db":"pubmed","term":args["q"],"retmax":args.get("n",10),"retmode":"json"})
    return [TextContent(type="text", text=r.text)]
# LangGraph: GEO study → DEG → enrichment → report
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class S(TypedDict):
    gse: str; degs: List[str]; pathways: List[str]; report: str

def fetch(s): s["degs"] = run_geo_deseq2(s["gse"]); return s
def enrich(s): s["pathways"] = enrichr(s["degs"]); return s
def write(s): s["report"] = llm_summary(s); return s

g = StateGraph(S)
g.add_node("fetch", fetch); g.add_node("enrich", enrich); g.add_node("write", write)
g.set_entry_point("fetch"); g.add_edge("fetch","enrich"); g.add_edge("enrich","write"); g.add_edge("write", END)
app = g.compile(checkpointer="sqlite:./chk.db")
app.invoke({"gse":"GSE161801"})
工作流

五、建立你自己的生醫 agent（6 步）

① 定義任務

明確輸入 / 輸出 / 成功標準。例：「給 GSE id → 輸出 DEG 表 + 富集 + markdown 報告」。

Pin down inputs / outputs / success criteria. E.g., "GSE id → DEG table + enrichment + markdown report."

② 識別工具

列出每一步要用的 API / package / 內部資料庫。

List the APIs / packages / internal DBs each step needs.

③ 包成 MCP server

把工具寫成 MCP server（或 LangGraph node），定義 input schema。

Wrap the tool as an MCP server (or LangGraph node); define its input schema.

④ 測試

給 3–5 個 known case 跑端到端，比對人工答案。

Run end-to-end on 3–5 known cases; compare with hand-curated answers.

⑤ 記錄

每一步 tool call、input、output、cost、latency 全留下；用 LangSmith / Helicone / 自寫 JSONL。

Persist every tool call, input, output, cost, latency — use LangSmith / Helicone / your own JSONL.

⑥ 審計

每月抽樣 10% trace 人工檢視；定期重跑 known case 偵測 regression。

Monthly: human-review 10% of traces; rerun known cases to detect regression.

⚠️
Agent 安全紅線：給 agent shell / DB 寫權限前，務必加 sandbox（容器、唯讀 mount、白名單 command）。一個能隨意 rm -rf 或寫 SQL DELETE 的 agent，會在迭代中變成事故來源。 Agent safety redline: Before giving an agent shell / DB write access, sandbox it (containers, read-only mounts, command whitelist). An agent free to rm -rf or run SQL DELETE will eventually become an incident.
真實案例

兩個深入案例：MCPmed 與 PubMed Slack alert

🔬

案例 1：MCPmed 把 GEO + UniProt + Ensembl 包成 MCP server，Claude 自主完成 cross-database 註釋

2025 年 Briefings in Bioinformatics 發表的 MCPmed 是首個系統性將生醫資料源包裝為 MCP server 的開源 initiative。研究者把 GEO（GEOquery）、UniProt REST API、Ensembl REST、PDB、Reactome、Open Targets 等各自寫成獨立 MCP server（每個 ~200 行 Python，遵守 MCP spec：tools/resources/prompts schema）。安裝後，Claude Desktop / Cursor / ChatGPT Connectors 在啟動時自動 discover 所有 server 提供的 tool。用戶在 Claude Desktop 問：「GSE171524 是什麼研究？取出 KRAS G12C inhibitor 治療組樣本，列出 differentially expressed proteins 並用 UniProt 註釋它們的 subcellular location，畫成 heatmap」——Claude 自主呼叫 geo_get_dataset → geo_extract_samples → uniprot_lookup（批次）→ python_exec（heatmap）四個 tool，全程 ~3 分鐘。關鍵技術貢獻：(1) 標準化解 schema，不同 client 通用；(2) 每個 server 獨立 maintain，不必 fork 大型 monolith；(3) 對研究者透明——他們不需要知道 API 細節。MCPmed 之後出現 Galaxy MCP、Bioconductor MCP、UCSC Genome Browser MCP 等延伸。

Case 1: MCPmed wrapped GEO + UniProt + Ensembl as MCP servers — Claude completed cross-database annotation autonomously

Published in Briefings in Bioinformatics (2025), MCPmed is the first systematic open-source effort to wrap biomedical data sources as MCP servers. The authors built independent MCP servers (~200 lines of Python each, following MCP spec: tools/resources/prompts schema) for GEO (GEOquery), UniProt REST, Ensembl REST, PDB, Reactome, Open Targets. Once installed, Claude Desktop / Cursor / ChatGPT Connectors auto-discover all tools at startup. A user asks in Claude Desktop: "What study is GSE171524? Pull KRAS G12C inhibitor-treated samples, list DE proteins, annotate their subcellular location via UniProt, and draw a heatmap." Claude autonomously calls geo_get_dataset → geo_extract_samples → uniprot_lookup (batched) → python_exec (heatmap) — done in ~3 minutes. Key contributions: (1) standardised schemas portable across clients; (2) each server maintained independently — no monolith fork; (3) transparent to researchers — they don't need to learn API details. MCPmed seeded downstream Galaxy MCP, Bioconductor MCP, UCSC Genome Browser MCP, etc.

📬

案例 2：某 lab 用 LangGraph 自動化每週 PubMed alert → 摘要 → Slack 推播

2025 年某 cardiology lab 想跟 KRAS / SOS1 / SHP2 三個 RAS pathway 的最新 paper，傳統流程：博後 NCBI MyBibliography 設 saved search、每週手動點 RSS、摘要寫 Slack——3 小時/週。改用 LangGraph 建 agent：節點包括 (1) pubmed_search（透過 Entrez API 抓上週新 paper）、(2) llm_rerank（Claude 對 ~50 個 hit 用 lab 興趣 prompt 排序前 10）、(3) llm_summarise（每篇 200 字摘要 + 對應現有 lab project 的關聯）、(4) slack_post（推播到 #literature 頻道，含 link 與 PMID）。Workflow 每週一 09:00 自動觸發；checkpoint 存 SQLite，失敗可 resume。執行 cost ~$0.30/週（Claude Haiku rerank + Sonnet summarise）。關鍵設計：(a) LangGraph 顯式 DAG 讓 PI 與博後可直接 review workflow；(b) 每步 LangSmith trace 可追蹤——若某週推送品質差，可 replay 該 trace；(c) human-in-the-loop interrupt：博後若覺得某篇值得深讀，可在 Slack 回 thumbs-up 觸發 deeper summary agent。教訓：production agent 不是「LLM + while loop」，而是「DAG + checkpoint + observability + 人類介入點」。lab 三個月後把 framework 開源為 litwatch。

Case 2: A lab used LangGraph to automate weekly PubMed alert → summary → Slack push

In 2025 a cardiology lab wanted to track new KRAS / SOS1 / SHP2 papers in the RAS pathway. Traditional flow: postdoc maintained an NCBI MyBibliography saved search, clicked RSS each week, summarised in Slack — 3 hr/week. They built a LangGraph agent with nodes: (1) pubmed_search (Entrez API, last week's papers); (2) llm_rerank (Claude scores ~50 hits with a lab-interest prompt, picks top 10); (3) llm_summarise (200-word summary + alignment with current lab projects); (4) slack_post (push to #literature with link + PMID). The workflow triggers Mondays 09:00; checkpoints persisted in SQLite for crash-resume. Run cost ~$0.30/week (Claude Haiku rerank + Sonnet summarise). Key design: (a) LangGraph's explicit DAG lets PI + postdoc review the workflow directly; (b) every step traced in LangSmith — bad weeks can be replayed; (c) human-in-the-loop interrupt: a thumbs-up in Slack triggers a deeper-summary agent. Lesson: production agents aren't "LLM + while loop" — they're "DAG + checkpointing + observability + human entry points." The lab open-sourced the framework as litwatch three months later.

常見誤區

使用 agent 最常踩的五個坑

⚠️
誤區 1：沒 audit log 讓 agent 亂跑
為什麼錯：agent 跑 100 個 tool call，沒 log 表示出問題時無法追蹤——是 tool call 失敗？LLM hallucinate？資料來源錯？「黑盒 agent」在 production 是事故來源。Anthropic 2024、OpenAI 2025 的 agent best-practice 都把 observability 列為 day-1 必需。
正確做法：用 LangSmith、Helicone、Arize Phoenix、自寫 JSONL log。每個 tool call 記 input/output/cost/latency；trace 可 replay；定期 sample 10% 人工 review。production agent 永遠 instrument 在先、deploy 在後。
Pitfall 1: Running agents without audit logs
Why wrong: 100 tool calls with no log means when something breaks you can't tell whether it's tool failure, LLM hallucination, or bad data. "Black-box agents" are production incident factories. Anthropic 2024 and OpenAI 2025 best-practice both put observability as a day-1 requirement.
Fix: use LangSmith, Helicone, Arize Phoenix, or roll your own JSONL log. Log every tool call's input/output/cost/latency; traces must be replayable; sample 10% for human review. Production agents are instrumented first, deployed second.
⚠️
誤區 2：Tool permission 過寬讓 agent 刪檔
為什麼錯：給 agent bash tool 即等同 root access；file_write 沒限制路徑可寫到 ~/.ssh/；db_query 沒區分 read/write 可跑 DELETE。2024 有公開案例 agent 在 debug 過程 rm -rf node_modules 不小心擴大到家目錄。
正確做法：(a) tool 設計遵守 principle of least privilege，read 與 write 分開、路徑白名單；(b) 把 agent 跑在 Docker container（唯讀 mount 主目錄、寫只允許 /workdir）；(c) destructive 操作（rm、DROP TABLE）強制 human-in-the-loop 確認；(d) Claude Code 提供 --dangerously-skip-permissions 才能 bypass——這個 flag 本身就是警告。
Pitfall 2: Over-broad tool permissions let agents delete files
Why wrong: giving an agent bash = root access; unbounded file_write can hit ~/.ssh/; db_query without read/write separation can run DELETE. In 2024 a public incident had an agent run rm -rf node_modules that escalated to the home directory.
Fix: (a) design tools with least privilege — separate read/write, whitelist paths; (b) run agents in Docker (read-only home mount, write only to /workdir); (c) destructive operations (rm, DROP TABLE) require human-in-the-loop confirmation; (d) Claude Code only bypasses with --dangerously-skip-permissions — the flag's name is the warning.
⚠️
誤區 3：無 budget cap 讓 agent 燒 100 倍 API cost
為什麼錯：agent loop 失控可能反覆呼叫 LLM 跑 1000+ iterations，cost 從預期 $0.50 變 $50；某學生在 hackathon 一夜跑掉 $400 是常見故事。Anthropic 與 OpenAI dashboard 都有 daily budget cap 設定。
正確做法：(a) provider 層設 daily/monthly cap；(b) agent 層設 max_iterations、max_tool_calls、max_cost_per_run；(c) trace 顯示 cumulative cost，超 threshold 觸發 alert；(d) loop detection——若連續 3 個 step 重複同一 tool call 且 input 高度相似，自動 break。LangGraph 的 recursion_limit 默認 25 是合理起點。
Pitfall 3: No budget cap lets agents burn 100× API cost
Why wrong: an out-of-control agent loop can run 1000+ iterations, blowing $0.50 into $50; the "$400 overnight at a hackathon" story is well known. Anthropic and OpenAI dashboards both expose daily budget caps.
Fix: (a) provider-level daily/monthly cap; (b) agent-level max_iterations, max_tool_calls, max_cost_per_run; (c) trace shows cumulative cost — alert above threshold; (d) loop detection — if three consecutive steps repeat the same tool with very similar inputs, auto-break. LangGraph's recursion_limit=25 default is a reasonable starting point.
⚠️
誤區 4：Loop 偵測沒做造成 infinite call
為什麼錯：agent 可能卡在「我需要更多資訊 → 呼叫 search → 仍不夠 → 再呼叫 search」迴圈，或 tool 報錯而 model 重試同一輸入 100 次。沒 loop detection 直接撞 max_iterations 才停，浪費 cost 與 latency。
正確做法：(a) tool input 做 hash，連續 N 次相同 hash 視為 loop 並 break；(b) 對 transient error（rate limit）有 exponential backoff，但對 deterministic error（404、bad query）直接報錯不重試；(c) 加 reflection node：每 5 step 讓 model 自己評估「是否在 loop」——這是 Reflexion (Shinn 2023) 的延伸。
Pitfall 4: Missing loop detection causes infinite calls
Why wrong: agents can get stuck "need more info → search → still not enough → search again" loops, or retry the same input 100 times on a tool error. Without loop detection only max_iterations stops them — wasting cost and latency.
Fix: (a) hash tool inputs; N consecutive identical hashes = loop, break; (b) exponential backoff for transient errors (rate limit), immediate failure for deterministic errors (404, bad query) — no retry; (c) add a reflection node every 5 steps to ask the model "am I looping?" — an extension of Reflexion (Shinn 2023).
⚠️
誤區 5：把 patient data 餵入未授權 agent
為什麼錯：個人版 ChatGPT / Claude / Gemini 不簽 BAA、預設可能用對話訓練模型；把含 PHI 的 EHR 表、含病人 ID 的 R Markdown 餵入 = HIPAA 違規。MCP server 若包了 hospital DB，client 端的 LLM provider 必須有 BAA。
正確做法：(a) PHI 必須走有 BAA 的 enterprise plan（Claude for Work, ChatGPT Enterprise, Azure OpenAI BAA）或 on-prem LLM；(b) agent 設計時 schema-level 標 sensitive field，自動脫敏；(c) IRB 審 agent workflow 與審 protocol 同步；(d) audit log 必含「哪個 user、哪份 data、跑了什麼 tool」以便事後追責。
Pitfall 5: Feeding patient data to non-authorised agents
Why wrong: personal ChatGPT / Claude / Gemini don't sign BAAs and may use conversations for training. Feeding PHI EHR tables or patient-ID R Markdown = HIPAA violation. If an MCP server wraps a hospital DB, the client-side LLM provider must have a BAA.
Fix: (a) PHI must run on BAA-covered enterprise plans (Claude for Work, ChatGPT Enterprise, Azure OpenAI BAA) or on-prem LLMs; (b) tag sensitive fields at the schema level and auto-redact; (c) IRB review of the agent workflow runs alongside protocol review; (d) audit logs must record "which user, which data, which tools" for accountability.
⚠️
誤區 6：迷信 multi-agent 比 single agent 更好
為什麼錯：CrewAI / AutoGen 等 multi-agent framework 在 demo 與 benchmark 上很漂亮，但 production 中 multi-agent 往往：(a) cost 3–10 倍；(b) latency 2–5 倍；(c) agent 之間溝通 noise 造成 cascade error；(d) debug 難度指數上升。Anthropic 2024 內部資料顯示 single agent + 良好 prompt + Reflexion 在大多 task 已足夠。
正確做法：先用 single agent + ReAct + 良好 system prompt；遇到 single agent 顯著失敗的 task 才考慮 multi-agent。複雜分工要驗證 cost-effectiveness（每多一個 agent 的邊際 accuracy gain vs 成本）。
Pitfall 6: Believing multi-agent beats single agent
Why wrong: CrewAI / AutoGen look great in demos and benchmarks, but in production multi-agent often: (a) costs 3–10×; (b) latency 2–5×; (c) inter-agent communication noise cascades errors; (d) debugging gets exponentially harder. Anthropic 2024 internal data: single agent + good prompt + Reflexion suffices for most tasks.
Fix: start with single agent + ReAct + a good system prompt; only escalate to multi-agent when single agent demonstrably fails. Validate cost-effectiveness: what marginal accuracy gain does each extra agent buy?
資深讀者深入

進階細節

🔬 MCP 三層架構、JSON-RPC 通訊、權限模型、agent reflection、cost monitoring

① MCP 的三層架構（host / client / server）。Host 是執行 LLM 的 process（Claude Desktop app、Cursor IDE）。Client 是 host 內負責跟單一 server 講話的子模組（一對多：一個 host 可有多個 client）。Server 是被連接的 process，暴露 tools（可呼叫 function）、resources（可讀資料如檔案 URI、DB row）、prompts（預設 prompt template）。設計哲學是「stateless server」——server 不記 conversation history，所有 context 從 client 傳來，使 server 可被多 client 並用。

② JSON-RPC 通訊機制。MCP 用 JSON-RPC 2.0 over stdio（local，子 process pipe）或 SSE / WebSocket（remote）。請求格式 {"jsonrpc": "2.0", "method": "tools/call", "params": {...}, "id": 1}，回應 {"jsonrpc": "2.0", "result": {...}, "id": 1}。主要 method 包括 initialize（capability negotiation）、tools/list、tools/call、resources/read、prompts/get、notifications/progress。spec 2025/11/25 stable，後續 backward-compatible。

③ MCP 權限模型。當前 MCP spec（2025）採「client controls user-facing consent」設計：client 顯示 server 提供的 tool list 給 user，user 在 tool call 時 approve / deny / always allow。Server 不直接信任 client；client 在 host 內被 sandbox。但實際安全仍依賴 host 實作品質——Claude Desktop 與 Cursor 在 2025 都有出現「accidentally always-allowed sensitive tool」的 bug。最佳實踐：host 應顯式 surface destructive tool（標紅）、要求每次 confirm；長期應期待 MCP spec 加入 OAuth-style scope。

④ Agent reflection / verification。Shinn et al. 2023（Reflexion, NeurIPS）：失敗後 model 寫自我批評進 memory，下次學。Madaan et al. 2023（Self-Refine）：iterative output critique。Yao et al. 2023（Tree of Thoughts, NeurIPS）：分支搜尋而非線性 ReAct。實務上：(a) verifier agent：獨立 LLM 跑「這個 answer 對嗎？」的二次審查；(b) self-consistency（Wang 2022）：跑 N 次取多數，agent 版本是 N 個獨立 trajectory 比較。Anthropic 2024 內部 data 顯示 verifier + self-consistency 在 high-stakes task 比 single trajectory 高 15–25%。

⑤ Cost monitoring 工程。(a) Provider 層：Anthropic / OpenAI dashboard 設 hard cap；(b) wrapper 層：LiteLLM proxy 或 OpenRouter 統一介面，可中央式 budget；(c) trace 層：LangSmith / Helicone / Arize Phoenix 每個 tool call 記 cost，匯總 dashboard；(d) cost-per-task：把 cost 對應到「完成 1 task」單位（如 1 paper summary = $0.05），便於 compare 不同 framework。對 production agent，cost-per-task 與 success-rate 是兩個並列的核心 KPI。

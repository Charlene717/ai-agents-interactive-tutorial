AI 加速資料分析

把 LLM 當資深 RA：解釋錯誤、寫 code、改 plot、解讀統計輸出、起草 methods，但別讓它替你選統計檢定。

Treat the LLM as a senior RA — let it explain errors, write code, polish plots, interpret stats, draft methods. Never let it pick your statistical test.

術語速查

資料分析 AI 必懂十詞

這 10 個詞貫穿這一章。先讀過一次，後面 EDA、missing data、causal inference 的討論就會輕鬆許多。

These ten words recur throughout. Skim once and the EDA / missing data / causal inference discussion below flows easily.

📓

Code Interpreter / Jupyter AI

ChatGPT Advanced Data Analysis（前身 Code Interpreter）與 JupyterLab AI 擴充：在 sandbox 跑 Python，自己看執行結果再下一輪 prompt。是「natural-language EDA」的代表，但 sandbox 沒持久檔案、有時 numerical instability。

ChatGPT Advanced Data Analysis (formerly Code Interpreter) and JupyterLab AI extensions: execute Python in a sandbox and feed results back to the next prompt. The poster child of natural-language EDA — but sandboxes lack persistent files and can have numerical instability.

🔍

EDA (Exploratory Data Analysis)

Tukey 1977 提出的「先看資料再選 model」工作流：missing 比例、distribution、outlier、變數間關係。LLM 加速 EDA 的 boilerplate（pandas-profiling、ydata-profiling），但「該看什麼」仍需研究者判斷。

Tukey's 1977 "look at your data before picking a model" workflow: missingness, distributions, outliers, pairwise relations. LLMs accelerate EDA boilerplate (pandas-profiling, ydata-profiling), but "what to look at" still requires the researcher.

🗂️

Schema

資料表的欄位名、型別、unit、constraint。給 LLM 寫 cleaning / merge code 前最該交付的「上下文最小集」。比直接貼整份 csv 更安全、更節省 token、避免幻覺欄位。

The column names, types, units, and constraints of a table. The minimal context you should give an LLM before asking for cleaning/merge code — safer than pasting the full CSV, saves tokens, and prevents hallucinated columns.

🐼

pandas / dplyr

Python pandas（Wes McKinney 2010）與 R dplyr（Wickham 2014）：兩大 tidy-data 操作 DSL。LLM 在兩者之間互轉幾乎無痛，但細節（pandas 的 SettingWithCopyWarning、dplyr 的 group_by 副作用）仍需注意。

Python pandas (Wes McKinney 2010) and R dplyr (Wickham 2014): the two tidy-data DSLs. LLMs convert between them nearly painlessly, but details (pandas SettingWithCopyWarning, dplyr group_by side effects) still need care.

📊

ggplot2

R 的 grammar-of-graphics 視覺化套件（Wickham 2007）。LLM 對 ggplot2 syntax 熟悉度極高，可一次到位「期刊 spec」。但 mis-leading axis、broken y、bar+error 等「figure 倫理問題」仍要人類把關。

R's grammar-of-graphics package (Wickham 2007). LLMs handle ggplot2 syntax superbly and can one-shot "journal spec." But misleading axes, broken y, bar+error bars and other figure-ethics issues still need human review.

🎯

SHAP

SHapley Additive exPlanations（Lundberg & Lee 2017）：把博弈論 Shapley value 套到 ML 解釋上，告訴你每個 feature 對單一 prediction 的貢獻。生醫 ML 的標配 explainability，但有 correlation 陷阱（Aas 2021）。

SHapley Additive exPlanations (Lundberg & Lee 2017): apply game-theoretic Shapley values to ML, telling you each feature's contribution to a single prediction. Standard biomedical-ML explainability, but with correlation pitfalls (Aas 2021).

🚨

Anomaly Detection

找「不像其他」的點：Tukey fence（IQR×1.5）、Z-score、Isolation Forest、autoencoder reconstruction error。LLM 可推薦方法，但「離群是錯誤還是真實生物學現象」仍是人類判斷。

Find points "unlike the rest": Tukey fence (IQR×1.5), Z-score, Isolation Forest, autoencoder reconstruction error. LLMs can recommend methods, but "is the outlier an error or real biology?" remains a human call.

❓

Missing Data (MCAR/MAR/MNAR)

Rubin 1976 分類：MCAR（completely at random）、MAR（at random given observed）、MNAR（not at random）。處理策略完全不同：MCAR 可 listwise delete；MAR 須 MI（multiple imputation, Rubin 1987）；MNAR 需 sensitivity analysis。AI 預設常做 listwise 是大坑。

Rubin 1976: MCAR (completely at random), MAR (at random given observed), MNAR (not at random). Strategies differ entirely: MCAR allows listwise deletion; MAR needs multiple imputation (Rubin 1987); MNAR needs sensitivity analysis. AIs default to listwise — a major pitfall.

🔗

Causal Inference

Pearl & Mackenzie 2018《The Book of Why》：從 association 到 causation 需要 DAG、do-calculus、IV（instrumental variable）。LLM 可寫 propensity score、IV 估計 code，但「哪個變數該 condition」必須由領域專家用 DAG 思考。

Pearl & Mackenzie 2018 (The Book of Why): moving from association to causation needs DAGs, do-calculus, and instrumental variables (IV). LLMs can code propensity scores and IV estimators, but "which variable to condition on" must come from a domain expert thinking with a DAG.

🪟

SuperPlots / Visual Honesty

Lord et al. 2020 (JCB)：用 dot-plot + 群組均值 + biological replicate 顯示比 bar+SEM 更誠實。Weissgerber 2015 (PLOS Bio)「Beyond Bar & Line Graphs」是同主題經典。LLM 寫 ggplot 容易產生 bar+SEM，要主動要求 SuperPlot 風格。

Lord et al. 2020 (JCB): dot-plot + group mean + biological replicates is more honest than bar+SEM. Weissgerber 2015 (PLOS Bio) "Beyond Bar & Line Graphs" is the classic. LLMs default to bar+SEM in ggplot — explicitly request SuperPlot style.

概覽

LLM 在資料分析中能做與不能做的事

2026 年，資料分析的瓶頸已不再是「會不會寫 ggplot2」，而是「能不能正確選 model、看懂 output、不過度詮釋」。LLM 在前者表現非常好，在後者依然容易出錯——尤其是「自動推薦統計檢定」常常忽略 assumption（normality、independence、proportional hazards）。

本章定義 5 個層級的「AI 可介入點」，給每層配對推薦工具，並用 DESeq2 / ggplot2 / pandas 三個生醫常見場景示範 prompt template。最後給一張「分流圖」讓你決定何時找統計師而不是 AI。

By 2026, the data-analysis bottleneck isn't "can you write ggplot2" — it's "can you pick the right model, read the output, and resist over-interpretation." LLMs are very good at the former, still error-prone at the latter — especially when "auto-suggesting" a statistical test, which often skips assumption checks (normality, independence, proportional hazards).

This chapter defines five levels of "AI can help here," pairs tools with each level, and walks through DESeq2 / ggplot2 / pandas prompt templates. A triage diagram tells you when to call a statistician instead.

💡
核心心法：把 LLM 當「資深 RA」——它寫 code 比你快、解釋語法比你穩，但研究問題、實驗設計、統計選型的責任永遠在你身上。給它 schema、給它 head() 輸出、給它明確問題，它的價值最大化。 Core mindset: Think of the LLM as a senior RA — faster at code, steadier at syntax, but research question, experimental design, and statistical choices remain your responsibility. Feed it the schema, head() output, and a precise question to maximise its value.

🎨

Not recommended to let AI choose t-test vs. Wilcoxon, paired vs. unpaired, mixed model or not — assumption checks belong to a human (docs / statistician).

案例 1：Hex AI 把 RNA-seq DESeq2 結果即時轉成 interactive notebook 讓 wet-lab PI 看懂

2025 年某 cancer immunology lab 跑完一個 anti-PD1 RNA-seq experiment（治療 vs control，每組 6 mice），bioinformatician 產出 DESeq2 results table（CSV，~20000 rows）丟給 wet-lab PI。PI 打開 CSV 看到 log2FoldChange、padj、baseMean 等欄位完全不知道從何下手——傳統做法是 bioinformatician 寫一份 PowerPoint，但 PI 一旦問「IFN gene set 是否 enriched？」、「能否依 KEGG pathway 分組？」就要重做。Lab 改用 Hex（hex.tech）——一個結合 Python notebook + AI assistant 的 collaborative platform：bioinformatician 上傳 DESeq2 CSV，用 AI assistant 一行 prompt「make a volcano plot with top 10 genes labelled, allow filtering by padj cutoff」即生成 interactive Plotly volcano。PI 自己拖 padj slider 從 0.05 到 0.01；點 IFN-gamma gene 看 raw counts boxplot；用 AI 問「show GSEA for these DEGs against MSigDB Hallmark」自動跑 fgsea + 可視化。整個 EDA 從「bioinformatician 來回 5 次」變成 PI 自助 30 分鐘。關鍵：Hex 的 AI assistant 不是 chatbot，而是 notebook-aware——它看到 cell output（dataframe shape、column types），生成的 code 馬上 run 並 render。對 wet-lab PI 把 BI tool 與 ML notebook 之間的 gap 補上。

Case 1: Hex AI turned RNA-seq DESeq2 results into a live interactive notebook for a wet-lab PI

In 2025 a cancer-immunology lab completed an anti-PD1 RNA-seq experiment (treatment vs control, 6 mice each). The bioinformatician produced a DESeq2 results CSV (~20k rows) and handed it off. The wet-lab PI opened it and stared at log2FoldChange, padj, baseMean — no idea where to start. Traditional route: a PowerPoint deck from the bioinformatician — but every PI follow-up ("is the IFN gene set enriched?" "can we group by KEGG pathway?") restarted the cycle. The lab switched to Hex (hex.tech), a collaborative notebook + AI-assistant platform. The bioinformatician uploaded the DESeq2 CSV and ran one prompt: "make a volcano plot with top 10 genes labelled and a padj cutoff filter." Hex emitted interactive Plotly. The PI dragged the padj slider from 0.05 to 0.01, clicked IFN-gamma to see raw-count boxplots, asked the AI "show GSEA for these DEGs against MSigDB Hallmark" — fgsea + visualization happened in-line. EDA went from "5 round-trips with the bioinformatician" to "30 minutes self-service." Key: Hex's AI assistant is notebook-aware, not a chatbot — it sees cell outputs (dataframe shapes, column types), and generated code runs and renders immediately. It closed the gap between BI tools and ML notebooks for wet-lab PIs.

🧬

案例 2：ChatGPT Advanced Data Analysis 用 4 個 prompt 完成 GTEx 子集 PCA + UMAP + 細胞型別 enrichment

2026 年初某 MD/PhD student 想快速看 GTEx v9 中 cardiovascular tissues（heart, aorta, coronary, atrial）的 expression structure，沒有寫 code 經驗。用 ChatGPT Advanced Data Analysis（前身 Code Interpreter）：(1) 「Read this GTEx subset CSV (~3000 samples × 20000 genes), give me schema and missing rate per column」——AI 在 sandbox 跑 pandas，回 summary；(2) 「Do PCA on log-CPM, plot PC1 vs PC2 colored by tissue」——AI 自動先 log1p + scale，產 matplotlib 圖；(3) 「Run UMAP on top 50 PCs, show three settings of n_neighbors」——AI 用 umap-learn 一次跑 3 個面板對比；(4) 「For the cluster in top-left, do over-representation analysis vs CellMarker DB」——AI 用 gseapy + Enrichr API 跑完，列出 cardiomyocyte markers significantly enriched。整個過程 ~45 分鐘，無一行 code 親手寫。關鍵注意：(a) sandbox 沒持久 file，每個 chat session 重來；(b) sandbox numerical 有時 unstable（同樣 PCA 跑兩次方向會反），需 prompt「set random_state=42」；(c) 不能上傳 PHI；(d) 適合 quick exploration，不適合 production pipeline（沒版本控制、沒 reproducibility）。學生後來把這個 EDA 結果交給 bioinformatician 跑正式版，省下 PI 與博後 ~2 週討論時間。

Case 2: ChatGPT Advanced Data Analysis ran PCA + UMAP + cell-type enrichment on a GTEx subset in 4 prompts

Early 2026, an MD/PhD student with no coding background wanted a quick look at GTEx v9 cardiovascular tissues (heart, aorta, coronary, atrial). Using ChatGPT Advanced Data Analysis (formerly Code Interpreter): (1) "Read this GTEx subset CSV (~3000 samples × 20000 genes), give schema and missingness per column" — AI ran pandas in the sandbox, returned summary; (2) "PCA on log-CPM, plot PC1 vs PC2 colored by tissue" — AI auto log1p + scaled and emitted matplotlib; (3) "UMAP on top 50 PCs, show three n_neighbors settings" — AI used umap-learn for a 3-panel comparison; (4) "For the top-left cluster, do over-representation analysis vs CellMarker DB" — AI ran gseapy + Enrichr API and listed cardiomyocyte markers as enriched. Key cautions: (a) sandbox files don't persist across sessions; (b) numerical instability (PCA direction can flip across runs) — prompt "set random_state=42"; (c) no PHI uploads; (d) good for exploration, bad for production (no version control, no reproducibility). The student handed the EDA to a bioinformatician for the formal run, saving ~2 weeks of PI/postdoc discussion time.

常見誤區

AI 加速資料分析最常踩的五個坑

⚠️
誤區 1：接受 AI 推薦的統計檢定卻不查 assumption
為什麼錯：AI 推薦 t-test 時不會主動跑 Shapiro-Wilk + Levene；推薦 Cox 不會跑 PH assumption test（Schoenfeld residuals）。若資料不符 assumption，p-value 完全失效。Knief & Forstmeier 2025 (R Soc Open Sci) 在生態學重新分析資料顯示「直接信 AI 推薦」可能有 ~30% 結論翻轉。
正確做法：對任何 stat test，先列 assumption 並逐一驗證；不滿足時用 robust 變體（Welch t-test 默認、Mann-Whitney、permutation）。Delacre 2017 建議 Welch 為 t-test 預設。
Pitfall 1: Accepting AI's stat-test recommendation without checking assumptions
Why wrong: when AI recommends a t-test it doesn't auto-run Shapiro-Wilk + Levene; for Cox it doesn't auto-run PH-assumption tests (Schoenfeld residuals). If assumptions fail, p-values are meaningless. Knief & Forstmeier 2025 (R Soc Open Sci) re-analysing ecology data showed "trusting AI's recommendation" flips ~30% of conclusions.
Fix: for any test, enumerate assumptions and verify each. If violated, use robust variants (Welch t-test as default, Mann-Whitney, permutation). Delacre 2017 argues Welch should be the t-test default.
⚠️
誤區 2：AI 自動處理 missing data 不分 MCAR/MAR/MNAR
為什麼錯：LLM 預設 missing 處理常是 listwise delete（dropna()）或 mean imputation——這在 MCAR 才正確。若是 MAR，需用 MI（multiple imputation, Rubin 1987）保留 uncertainty；MNAR 須 sensitivity analysis（如 Heckman selection）。在 EHR、survey、long-term cohort 上 MNAR 極常見，listwise 會嚴重 bias。
正確做法：先做 missing pattern 視覺化（mice::md.pattern, naniar::vis_miss）；用 Little's MCAR test 或檢視 missing 與其他變數的關聯；MAR 用 mice / Amelia / IterativeImputer + Rubin's rules；MNAR 用 multiple sensitivity scenarios。
Pitfall 2: AI handling missing data without distinguishing MCAR/MAR/MNAR
Why wrong: LLMs default to listwise deletion (dropna()) or mean imputation — valid only under MCAR. MAR requires multiple imputation (Rubin 1987) to preserve uncertainty; MNAR needs sensitivity analysis (e.g., Heckman selection). In EHR, surveys, and long cohorts, MNAR is the norm — listwise badly biases results.
Fix: first visualise the missingness pattern (mice::md.pattern, naniar::vis_miss); run Little's MCAR test or check missingness vs other variables; MAR → mice / Amelia / IterativeImputer + Rubin's rules; MNAR → multiple sensitivity scenarios.
⚠️
誤區 3：AI 把 categorical 當 ordinal（或反之）
為什麼錯：blood type（A/B/AB/O）、ICD code、tissue type 是 nominal categorical；NYHA Class（I/II/III/IV）、cancer stage 是 ordinal。LLM 看到「Class 1, 2, 3, 4」常以為是 numeric 直接 fit linear regression——這把 Class III - Class I 強加為 2 倍的「Class II 距離」，毫無生物意義。
正確做法：給 LLM schema 時明確標 dtype（nominal/ordinal/numeric）；fit 前用 R 的 factor() + levels = c(...) 或 pandas Categorical(ordered=True)；對 ordinal 用 ordinal regression（MASS::polr、statsmodels OrderedModel）而非 OLS。
Pitfall 3: AI treats categorical as ordinal (or vice versa)
Why wrong: blood type (A/B/AB/O), ICD codes, tissue types are nominal; NYHA Class (I–IV), cancer stage are ordinal. LLMs see "Class 1, 2, 3, 4" and often default to numeric linear regression — forcing "Class III − Class I = 2 × Class II distance," biologically meaningless.
Fix: when giving the LLM a schema, explicitly tag dtype (nominal/ordinal/numeric); before fitting use R's factor() + levels = c(...) or pandas Categorical(ordered=True); for ordinal data use ordinal regression (MASS::polr, statsmodels OrderedModel), not OLS.
⚠️
誤區 4：ggplot 美化但 mis-leading axis
為什麼錯：AI 一鍵美化 ggplot 容易：(a) y-axis broken / zoom-in 放大 5% 差異看起來像 10 倍；(b) bar+SEM 隱藏 individual variance；(c) cherry-pick 配色讓 control 看起來灰、treatment 看起來「漂亮的紅」——這些都是 figure ethics 問題。Weissgerber 2015 (PLOS Bio) 與 Lord 2020 (JCB) 已提供 best practice。
正確做法：(a) y 軸從 0 開始（除非有強理由）；(b) 用 SuperPlot（dot + group mean + biological replicate）而非 bar+SEM；(c) 用 perceptually-uniform palette（viridis、Okabe-Ito）；(d) 對任何 figure 自問「reviewer 會說我在誤導嗎？」。
Pitfall 4: Polished ggplots with misleading axes
Why wrong: AI one-shot polishing easily produces: (a) broken/zoomed y-axis making a 5% difference look 10×; (b) bar+SEM hiding individual variance; (c) cherry-picked palettes making control look grey and treatment a "pretty red" — all figure-ethics violations. Weissgerber 2015 (PLOS Bio) and Lord 2020 (JCB) provide best practice.
Fix: (a) y-axis starts at 0 unless strongly justified; (b) use SuperPlot (dot + group mean + biological replicate) instead of bar+SEM; (c) use perceptually-uniform palettes (viridis, Okabe-Ito); (d) for any figure, ask "would a reviewer say I'm misleading?"
⚠️
誤區 5：在 nested data 直接 lm() 忽略 random effects
為什麼錯：scRNA-seq 多細胞屬於同一病人、multi-site clinical trial 多病人屬於同一醫院、repeated-measure ANOVA 多 timepoint 屬於同一受試——資料有 nested / clustered 結構但 lm() 假設 independence，會嚴重低估 SE、誇大 p-value。LLM 看到 long-format 表常直接 lm，忽略 cluster。Bolker 2009 (TREE) 是經典 reference。
正確做法：對任何 nested data 用 mixed model（lme4::lmer、glmmTMB、Stan brms）。scRNA-seq 用 pseudobulk（per-sample 平均後 DESeq2）或 muscat::pbDS；EHR 用 GLMM with site random effect。
Pitfall 5: Running lm() on nested data ignoring random effects
Why wrong: scRNA-seq cells belong to patients, multi-site trials cluster within hospitals, repeated-measure ANOVA shares subject-timepoints. The data is nested/clustered but lm() assumes independence — severely under-estimating SE and inflating p-values. LLMs see long-format tables and default to lm without clustering. Bolker 2009 (TREE) is the classic reference.
Fix: any nested data → mixed model (lme4::lmer, glmmTMB, Stan brms). scRNA-seq → pseudobulk (per-sample mean, then DESeq2) or muscat::pbDS; EHR → GLMM with site random effect.
⚠️
誤區 6：AI 寫 ML pipeline 沒做 data leakage 防護
為什麼錯：常見 leakage 來源——(a) 在 train/test split 前做 normalisation / feature selection；(b) 同一個 patient 的多次 sample 被分到 train 與 test；(c) 用未來資料預測過去（label peek）；(d) 用 cross-validation 跑 hyperparameter 又用同一份 CV 報 performance。Kapoor & Narayanan 2023 在 medical ML 重新分析發現 ~50% 論文有 leakage。
正確做法：(a) preprocessing 包入 Pipeline / recipes，只在 train fold fit；(b) GroupKFold 對 patient ID 分組；(c) nested CV：outer 報 performance、inner 選 hyperparameter；(d) 永遠 held-out 一份 test set，最後一次評估。
Pitfall 6: AI-written ML pipelines lacking data-leakage protection
Why wrong: common leakage sources — (a) normalisation / feature selection before train/test split; (b) one patient's samples split across train and test; (c) using future data to predict past (label peeking); (d) tuning hyperparameters on CV and reporting performance from the same CV. Kapoor & Narayanan 2023 found ~50% of medical-ML papers have leakage.
Fix: (a) wrap preprocessing in sklearn Pipeline / tidymodels recipes, fit only on train folds; (b) GroupKFold by patient ID; (c) nested CV: outer reports performance, inner tunes; (d) always hold out a final test set for one-shot evaluation.

進階細節

🔬 AI 對 causal inference 的局限、SHAP 解釋陷阱、Tufte 原則

① AI 對 causal inference 的根本局限。LLM 學的是 statistical association（co-occurrence），但 causal inference 要求 distribution interventional（do-calculus, Pearl 2009）。Schölkopf 2021（Toward Causal Representation Learning, IEEE）證明從觀察資料學「invariant causal mechanism」是 underdetermined——除非加入額外假設（faithfulness、no hidden confounder）或 randomized experiment。LLM 寫 propensity score / IV / DiD code 都沒問題，但「該 condition 哪些變數、IV 是否 valid、parallel trend 是否成立」必須由 DAG 思考的領域專家判斷。Cinelli & Hazlett 2020（OMNI, JRSS-B）的 sensitivity analysis 工具可以量化 unmeasured confounder 對結論的威脅。

② SHAP 解釋的陷阱（Aas et al. 2021, AI）。原始 SHAP 假設 feature 互相獨立計算 Shapley value。實際生醫資料常有強 correlation（age + Charlson、BMI + waist circumference）。在 correlated features 下 SHAP 會把 importance 「分攤」到 correlated 兄弟 features，看起來「每個都不重要」。Aas 2021 提出 conditional SHAP（給定其他 features 的條件分布）修正，但成本是計算 expensive。對醫師：看 SHAP plot 時要問「這些 features 之間 correlation matrix 長怎樣？」

③ AI-generated plots 與 Tufte 原則。Tufte 1983（The Visual Display of Quantitative Information）核心：data-ink ratio 高、minimize chartjunk、small multiples 比 stacked 好。AI 預設 ggplot 容易違反：(a) gridline 過密；(b) legend 重複資訊；(c) 3D pie chart（仍偶有出現）。對應 fix：theme_classic()、guides(color = "none")、用 small-multiple faceting。生醫 figure 還有 Weissgerber 2015 (PLOS Bio) 補充「show individual data points」、Lord 2020 (JCB) 的 SuperPlot 模式。

④ Hex AI / Code Interpreter 的工程限制。(a) Sandbox 沒持久檔案，每 session 重來；(b) memory 上限 ~1 GB，大 dataset 須先 subsample 或用 polars/Arrow；(c) numerical instability—— 同一個 PCA 跑兩次可能 sign flip（PC1 方向反），須設 random_state；(d) 不能直接 access 外部 DB / API（除非 Code Interpreter 開放 internet）；(e) Hex / Mode / Observable 等 commercial platform 有 secrets manager 與 BAA，個人版 ChatGPT 沒有。

① The fundamental AI vs causal-inference gap. LLMs learn statistical association (co-occurrence), but causal inference requires interventional distributions (do-calculus, Pearl 2009). Schölkopf 2021 (Toward Causal Representation Learning, IEEE) showed that learning "invariant causal mechanisms" from observation alone is underdetermined — extra assumptions (faithfulness, no hidden confounder) or randomized experiments are required. LLMs happily write propensity-score / IV / DiD code, but "which variables to condition on, whether the IV is valid, whether the parallel-trends assumption holds" must come from a domain expert thinking with a DAG. Cinelli & Hazlett 2020 (OMNI, JRSS-B) provide sensitivity-analysis tools to bound the threat from unmeasured confounders.

② SHAP pitfalls (Aas et al. 2021, AI). Vanilla SHAP assumes feature independence when computing Shapley values. Real biomedical features are often strongly correlated (age + Charlson, BMI + waist circumference). Under correlation, SHAP splits "importance" across correlated siblings, making each look unimportant. Aas 2021 proposed conditional SHAP (using conditional feature distributions) as a fix, at heavy compute cost. For clinicians: always ask "what does the correlation matrix among these features look like?" before reading a SHAP plot.

③ AI plots and Tufte's principles. Tufte 1983 (The Visual Display of Quantitative Information): high data-ink ratio, minimize chartjunk, small multiples beat stacked. AI default ggplots tend to violate: (a) over-dense gridlines; (b) duplicated legend info; (c) the occasional 3D pie chart. Fixes: theme_classic(), guides(color = "none"), small-multiple faceting. For biomed figures add Weissgerber 2015 (PLOS Bio) "show individual data points" and Lord 2020 (JCB) SuperPlots.

④ Engineering limits of Hex AI / Code Interpreter. (a) Sandboxes don't persist files; each session resets. (b) ~1 GB memory cap; large datasets need subsampling or polars/Arrow. (c) Numerical instability — PCA can sign-flip across runs; set random_state. (d) No external DB/API access (unless internet enabled). (e) Hex / Mode / Observable etc. have secrets management and BAA; personal ChatGPT doesn't.

討論問題

與你的 lab 思辨這四題

Q1: AI 能否 ever 取代統計師？

2026 年 LLM 已能寫各種 stat code、解釋 output、甚至自動跑 assumption check。某 lab PI 提案：「我們不再聘專職統計師，用 AI + 博後審即可」。lab 內統計師反駁：「AI 會說『這是 mixed model』，但不會問『你的隨機 effect 是 site 還是 patient？這影響 inference 對象』。」請思考：哪些統計問題 AI 已足夠？哪些必須人類統計師？「AI + 統計師」協作模式該怎樣設計？clinical trial 的 SAP 是否可由 AI 起草？

Q1: Can AI ever replace a statistician?

By 2026 LLMs write stat code, interpret output, and even auto-check assumptions. A PI suggests: "We don't need a full-time statistician — AI + postdoc review will do." The lab statistician counters: "AI will say 'this is a mixed model' but won't ask 'is your random effect site or patient? It changes the inference target.'" Consider: which statistical tasks are AI-sufficient? Which require a human statistician? How should "AI + statistician" collaboration be structured? Should an AI draft the SAP for a clinical trial?

Q2: 黑盒分析 vs 透明分析在 reviewer 面前哪個更可信？

傳統 SPSS / R 分析提供完整 syntax 與 output，reviewer 可重跑。AI assistant 分析（Hex AI、ChatGPT ADA）是 conversational：「我問它它跑」，最終 figure 來自一連串對話，無法逐字重現。某 NEJM reviewer 拒絕一篇用 ChatGPT 完成 EDA 的稿件：「我看不到 deterministic pipeline」。請思考：對 reviewer 與 reader，conversational analysis 該如何揭露？是否該附 chat transcript？AI 分析的 reproducibility 標準（Wilkinson 2016 FAIR）該如何制定？

Q2: Black-box vs transparent analysis — which is more trustworthy to reviewers?

Traditional SPSS / R analyses ship full syntax + output reviewers can rerun. AI-assistant analyses (Hex AI, ChatGPT ADA) are conversational — "I ask, it runs" — and the final figure emerges from a dialogue, not reproducible verbatim. One NEJM reviewer rejected an AI-EDA paper: "I can't see a deterministic pipeline." Consider: how should conversational analyses be disclosed to reviewers and readers? Should chat transcripts be attached? What reproducibility standard (Wilkinson 2016 FAIR) should AI analyses follow?

Q3: 訓練 wet-lab PI 用 ChatGPT ADA 還是 Hex AI？

某 cancer center 想推「PI 自助分析」訓練：ChatGPT Advanced Data Analysis 入門低、無安裝、free tier 即可開始；Hex 是 notebook-aware、可 share、有 BAA。但 ChatGPT ADA sandbox 限制多（無 file 持久、PHI 風險）；Hex 學習曲線較高、需訂閱。請思考：對沒寫 code 經驗的 wet-lab PI，第一個工具該推什麼？訓練該強調哪些素養（schema 思維、assumption check、reproducibility）？lab 預算有限時的 cost-benefit？

Q3: Train wet-lab PIs on ChatGPT ADA or Hex AI?

A cancer center wants to train "PI self-service analysis": ChatGPT Advanced Data Analysis is low entry, no install, free tier; Hex is notebook-aware, shareable, has BAA. But ChatGPT ADA sandboxes have constraints (no file persistence, PHI risk); Hex has a steeper learning curve and a subscription. Consider: for a PI with no coding background, which is the first tool to recommend? Which literacy skills should training emphasize (schema thinking, assumption checks, reproducibility)? Cost–benefit on a tight budget?

Q4: AI 揭露在 figure-level 的標準？

某論文 figure 1 是 Excel 手繪 schematic，figure 2 是 PI 寫的 ggplot，figure 3 是博後用 Cursor 寫的 ggplot，figure 4 是博後用 ChatGPT ADA 一句 prompt 生成的 volcano。reviewer 問「哪些是 AI 生成？」博後不確定該逐 figure 揭露還是論文 methods 概括。ICMJE 2024 要求 AI 揭露但未指定粒度。請思考：揭露粒度該到哪？「AI 寫 vs AI 美化 vs AI 設計」如何分類？是否該在 figure caption 加 tag（如「[AI-assisted: ChatGPT-4o, 2026/03]」）？

Q4: Figure-level granularity for AI disclosure?

In one paper Figure 1 is an Excel hand-drawn schematic, Figure 2 a PI-written ggplot, Figure 3 a postdoc + Cursor ggplot, Figure 4 a one-prompt ChatGPT ADA volcano. A reviewer asks "which are AI-generated?" The postdoc isn't sure whether to disclose per figure or summarise in methods. ICMJE 2024 requires AI disclosure but doesn't specify granularity. Consider: how granular should disclosure be? How do you categorise "AI wrote vs polished vs designed"? Should figure captions carry tags (e.g., "[AI-assisted: ChatGPT-4o, 2026/03]")?
